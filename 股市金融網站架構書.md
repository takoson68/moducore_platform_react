# 股市金融網站架構書

## 1. 文件目的

本文件用於整理「研究型股市金融網站」的新專案前端架構方向。

重點如下：

- 本文件只做架構整理，不做功能實作
- 不修改既有平台層核心
- 不修改既有 world、container、router 核心機制
- 新功能應以「新專案 / 新 project / 新 module」方式落地
- 新專案自己的需求，應盡量由新專案自己的模組實作，不污染平台通用層

---

## 2. 本次已確認的邊界原則

### 2.1 不可任意變動的平台核心

以下層級屬於平台核心或全域基礎設施，未經明確同意不應修改：

- `platform/frontend/src/react/`
- `platform/frontend/src/app/container/`
- `platform/frontend/src/app/stores/`
- `platform/frontend/projects/loadProject.js`
- `platform/frontend/projects/moduleDiscovery.js`
- `platform/frontend/projects/moduleRegistryFactory.js`
- `platform/frontend/projects/modulesRegistry.js`
- `_VibeCore/` 下的 world、boundary、convention 文件

這些內容屬於平台共用能力，不應為了單一新專案需求而調整。

### 2.2 新專案應優先採用的方式

新股市金融網站應採用以下原則：

- 建立獨立 project
- 在該 project 下建立自己的 layout、styles、modules、docs
- 需要的新頁面、新互動、新 mock data、新 UI 元件，優先放在 project module 內
- 若未來要接 API，應放在 module-local `service.js` 或 `api/`
- 若未來要做研究頁、觀察清單頁、比較頁，也應優先以 module 為單位擴充

## 3. 目前平台前端結構理解

依照目前 repo 結構，前端平台採用以下模式：

- `project` 決定目前載入哪一個專案
- `project.config.js` 宣告專案資訊與要啟用的 modules
- `modules/` 底下每個 module 自己註冊 routes、stores、panels、services
- React runtime 透過 world 啟動並掛載 route page
- layout 由 project 自行提供

### 3.1 啟動流程

目前觀察到的流程大致如下：

1. `src/main.js` 啟動前端
2. `projects/loadProject.js` 依 `VITE_PROJECT` 載入對應 `project.config.js`
3. `src/react/startReactApp.jsx` 啟動 React app
4. `src/react/reactWorld.js` 建立 runtime world
5. `src/react/reactBoot.js` 做 module discovery 與 install
6. project layout 決定頁面殼層
7. route 對應到 module page

### 3.2 Project 結構模式

目前專案結構模式已相對清楚：

```txt
platform/frontend/projects/<project-name>/
  project.config.js
  layout/
  modules/
  styles/
  docs/
```

### 3.3 Module 結構模式

目前 module 的典型結構如下：

```txt
modules/<module-name>/
  components/
  pages/
  routes.js
  store.js
  index.js
  service.js        # 視需要
  api/              # 視需要
  utils/            # 視需要
```

## 4. 對新股市金融網站的架構建議

## 4.1 建議以新 project 建置

本次需求不是在既有驗證型 project 上直接擴充比較合適，而是應建立新的股市金融 project。

建議型態：

- project 名稱先獨立，例如 `stock-research`
- project 擁有自己的：
  - `project.config.js`
  - `layout/`
  - `modules/`
  - `styles/`
  - `docs/`

這樣可以避免：

- 影響既有示範型 project
- 讓股市金融領域需求混入平台驗證案例
- 為了新專案需求去改平台層

## 4.2 建議模組切分方向

依照目前產品定位，未來可優先切成以下模組：

- `watchlist`
  觀察清單主頁
- `research`
  個股研究頁
- `comparison`
  多來源比較與同業比較
- `events`
  財報、法說、除權息事件整合
- `export-center`
  匯出與研究紀錄

第一版若只做主畫面，可先從 `watchlist` 模組開始。

## 5. 觀察清單頁的建議落點

若未來開始實作，「觀察清單頁」最適合放在：

- 新 project 的 `modules/watchlist/`

而不是：

- 平台共用層
- world 層
- container 核心層
- 既有非股市專案 module

### 5.1 建議路由

建議作為新 project 的主頁之一，路由可規劃為：

- `/watchlist`

若它是第一版核心首頁，也可視新 project 的 layout 決定是否作為預設進入頁。

## 6. 目前已辨識出的可沿用能力

本次掃描後，可沿用的是「結構與模式」，不是直接共用一套成熟商業元件庫。

### 6.1 可沿用的結構能力

- project 載入機制
- module discovery 機制
- route 註冊機制
- world 啟動流程
- module store 註冊模式
- project layout 模式
- page 與 panel 的分工方式

### 6.2 可沿用的 UI 骨架

現有專案中可參考的，是頁面殼層與基本樣式模式，例如：

- `react-page`
- `react-page__header`
- `react-panel-grid`
- `react-card`
- `react-actions`
- `react-status-list`

這些可作為新 project 初期參考，但建議新股市金融 project 自己維護自己的樣式與頁面語彙。

### 6.3 目前沒有現成可直接重用的能力

掃描結果顯示，目前未看到成熟可直接套用的：

- 清單切換 UI 元件
- 表格 UI 元件
- 拖曳排序套件或封裝
- 狀態 badge 元件
- 股市研究摘要卡元件
- 匯出 CSV 專用模組

因此新專案若要實作，應由新專案 module 自行建立。

## 7. 觀察清單頁第一版的功能分區建議

依照目前需求，這一頁可固定拆成 5 個區塊：

1. 清單切換
2. 股票列表
3. 狀態標記
4. 快速摘要
5. 匯出按鈕

### 7.1 清單切換區

第一版應支援：

- 顯示目前清單
- 切換清單
- 建立新清單
- 刪除清單
- 複製清單

### 7.2 股票列表區

第一版應支援：

- 顯示股票列表
- 新增股票
- 移除股票
- 進入研究頁
- 拖曳排序

### 7.3 狀態標記區

每檔股票狀態：

- 觀察中
- 已買
- 已賣
- 目標

並支援依狀態篩選。

### 7.4 快速摘要區

至少應包含：

- 代號
- 名稱
- 目前狀態
- 簡短結論
- 異常標記

### 7.5 匯出區

第一版建議：

- 匯出整份清單
- 依狀態篩選匯出
- 僅做 CSV

## 8. 新專案資料層建議

在沒有後端前，建議先採以下分層：

### 8.1 畫面狀態層

由 module 的 `store.js` 管理：

- 目前選擇的清單
- 清單集合
- 股票排序
- 狀態篩選
- 匯出條件
- 使用者當前操作結果

### 8.2 Mock 資料層

由 module-local 檔案管理，例如：

- `mockData.js`
- `utils/anomaly.js`

用途：

- 提供初始觀察清單
- 提供股票基本資料
- 提供快速摘要欄位
- 提供異常標記規則

### 8.3 未來 API 接點

未來若接真實 API，建議新增：

- `service.js`
- `api/`

原則如下：

- API 回傳資料先做 normalize
- page 不直接碰 API 細節
- store 不直接承擔 API protocol 細節
- module 自己管理自己的資料轉換

---

## 9. 對異常標記與匯出的架構建議

### 9.1 異常標記

第一版可以先是 mock 規則，不必綁死真實資料來源。

例如：

- 多來源價格差異過大
- 更新時間差異過大
- 指標計算口徑不同

建議將規則獨立在 module utils，而非直接寫死在頁面 JSX。

### 9.2 匯出功能

第一版匯出建議維持純前端 CSV：

- 將目前列表資料轉成 CSV
- 支援整份匯出
- 支援依狀態篩選後匯出

同樣建議抽出為 module-local util，避免把匯出邏輯塞進 page。

---

## 10. 風險與假設

### 10.1 風險

- 目前平台沒有現成的拖曳封裝，第一版若要做排序，需要新專案自己處理
- 目前平台沒有股市專用 UI 套件，欄位顯示與互動要自行設計
- 若直接在既有 project 上做，容易與示範專案目的混在一起
- 若動到平台共用層，會提高未來維護風險

### 10.2 假設

- 新股市金融網站會是獨立 project，不是既有示範 project 的小修改
- 第一版可接受 mock data
- 研究頁在第一階段可能仍為 placeholder
- 平台層只提供啟動與掛載規則，不承擔股市領域邏輯

---

## 11. 本次結論

本次整理後，架構上最重要的結論如下：

1. 新股市金融網站應建立為獨立 project
2. 觀察清單頁應作為新 project 的 module 實作
3. 不應為了新專案需求去修改平台核心層
4. 可沿用的是 project/module/router/store 的結構模式
5. 不足的 UI 與商業邏輯能力，應由新專案自行建立
