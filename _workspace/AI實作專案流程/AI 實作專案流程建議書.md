# AI 實作專案流程建議書（MVP + Gate 驗收版）

---

## 🎯 文件目的

建立一套可重複使用、可驗收、可收斂的 AI 協作專案流程。  
避免開發過程發散、方向錯誤、最後重修。

---

# Phase 1 — 需求蒐集（Problem Definition）

## 目標
確認「專案解決什麼問題」與「成功標準是什麼」。

## 必做輸出

- 使用者角色（Persona）
- 使用者故事（User Story）
- 需求清單（Must / Should / Could）
- 成功衡量標準（可量化）

## User Story 格式

> 身為（誰）  
> 我希望（做什麼）  
> 以便於（達成什麼價值）

## Gate（完成標準）

- 所有 Must 需求明確定義
- 至少寫出 1 條可量化核心成功指標

---

# Phase 2 — 需求分析（MVP 定義）

## 目標

將需求收斂成「最小可行版本」。

## 必做輸出

- MVP 範圍定義
- 功能地圖（Feature Map）
- 頁面清單（Page List）
- 核心流程（Happy Path）
- 資料模型草圖（Entity 定義）
- 權限與狀態流程

## Gate

- 可描述使用者從進站到完成任務的完整流程
- 可說清楚有哪些資料模型（DB / Store）

---

# Phase 3 — 技術與架構配置

## 目標

避免技術選型混亂。

## 必做輸出

- 技術棧選擇
- 資料儲存策略
- API 設計原則
- 部署策略
- 環境設定（dev / prod）
- 測試策略（最小可行測試）

## Gate

- 專案啟動方式明確
- 資料流向明確
- API 邊界清楚

---

# Phase 4 — 建立基礎架構（Skeleton）

## 目標

產出可執行骨架專案。

## 必做輸出

- 基本目錄結構
- 路由系統可運作
- API 假資料可運作
- 基本狀態管理架構
- Lint / 格式規範

## Gate

- 專案可正常啟動
- 頁面可切換
- API 可正常回傳資料

---

# Phase 5 — 垂直切片開發（Vertical Slice）

## 目標

一次完成一個完整功能。

## 原則

一次打通：

UI → State → API → 儲存 → 回應

## 每次必做輸出

- 完整可運作功能
- 手動驗收步驟
- 預期結果說明

## Gate

- 功能可完整操作
- 已處理邊界條件
- 無 console error

---

# Phase 6 — 測試與修正

## 目標

強化穩定性。

## 必做項目

- 邊界條件測試
- 錯誤流程測試
- 空資料測試
- 權限測試

## 輸出

- Bug 清單
- 修正紀錄

## Gate

- 核心流程無阻塞
- 不影響既有功能

---

# Phase 7 — 收斂與交付

## 目標

讓專案具備展示與交付能力。

## 必做輸出

- README（如何啟動、測試帳號）
- 功能完成清單
- 已知限制
- 下一步 Roadmap

## Gate

- 第三方可依文件成功啟動
- 可完整 demo 核心流程

---

# 核心流程總結

1️⃣ 需求定義  
2️⃣ MVP 收斂  
3️⃣ 技術配置  
4️⃣ Skeleton  
5️⃣ 垂直切片開發  
6️⃣ 測試穩定  
7️⃣ 收斂交付  

---

# 最重要規則

- 每個 Phase 必須有 Gate
- 未達 Gate 不得進入下一階段
- 每次只讓 AI 產出當前階段成果
- 禁止跨階段實作

---

# 建議用法

當與 AI 協作時：

- 指定當前 Phase
- 要求只產出該 Phase 的文件
- Gate 通過才進入下一階段
