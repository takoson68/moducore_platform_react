# 草稿與正式資料格式

## 文件定位
本文件用於定義餐廳地圖產生器的草稿格式、正式儲存格式與兩者之間的轉換原則。

## 1. 原則

- 草稿格式用於恢復編輯狀態。
- 正式格式用於穩定載入與對外交換。
- 內部狀態格式與交換格式可不同，但語意需一致。
- 轉換層負責整理內部狀態與對外格式。

## 2. 草稿格式

草稿資料至少包含：

- `version`
- `map`
- `draftState`
- `savedAt`

範例：

```json
{
  "version": 1,
  "map": {
    "id": 1,
    "name": "一樓內用區",
    "width": 1200,
    "height": 800,
    "objects": []
  },
  "draftState": {
    "mode": "map",
    "tool": "polyline",
    "activeObjectId": null,
    "pendingPolyline": [
      { "x": 100, "y": 100 },
      { "x": 300, "y": 100 }
    ]
  },
  "savedAt": "2026-03-17T10:00:00+08:00"
}
```

## 3. 正式格式

正式資料至少包含：

- `version`
- `map`

範例：

```json
{
  "version": 1,
  "map": {
    "id": 1,
    "name": "一樓內用區",
    "width": 1200,
    "height": 800,
    "objects": [],
    "tables": []
  }
}
```

## 4. 轉換原則

- 由內部 editor state 轉草稿格式時，可帶入 `draftState`。
- 由內部 editor state 轉正式格式時，只保留已完成資料。
- 未完成 `Polyline` 不直接寫入正式資料。
- `tables` 尚未進入桌位階段前，可不寫入正式資料。

## 5. 對接原則

- 櫃台模組匯入時，以正式格式為主。
- 若匯入草稿資料，需先經轉換層整理為櫃台可用格式。
- 與既有資料庫對接時，不直接使用 editor state 作為資料庫結構。
