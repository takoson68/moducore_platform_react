# 模組哲學改善執行書

更新日期：2026-03-10

## 1. 目的

本文件將 [模組哲學健檢書](/F:/GitHub/moducore_platform/platform/frontend/projects/dineCore/docs/模組哲學健檢書.md)
轉成可直接執行的工程文件。

本文件只處理以下目標：

1. 收斂目前最明確違反模組哲學的結構
2. 降低顧客流程模組群的直接耦合
3. 讓 `LayoutRoot` 回到 project shell 應有的責任邊界

本文件不處理：

1. 新功能追加
2. 視覺重設計
3. 非必要的全域重構

---

## 2. 執行原則

### 2.1 唯一診斷來源

本次改善的診斷基礎只採用：

- [模組哲學健檢書.md](/F:/GitHub/moducore_platform/platform/frontend/projects/dineCore/docs/模組哲學健檢書.md)

若過程中發現新的偏離：

- 先回寫健檢判定
- 再決定是否納入當前階段

### 2.2 最小合法變更

每一階段只允許做當前階段需要的最小合法變更。

不得：

1. 順手改 unrelated 模組
2. 順手換 UI 架構
3. 順手重命名大量檔案

### 2.3 偏離修正優先

若本階段改動本身又造成新的模組哲學偏離：

1. 先修正偏離
2. 再宣告本階段完成

不得把新偏離留給下一階段。

---

## 3. 固定健檢尺標

每一階段結束時，必須固定檢查以下項目：

1. 是否直接知道他模組 store 名稱
2. 是否直接知道他模組 session 結構
3. 是否直接知道他模組資料 shape
4. 是否可在不改他模組內部的前提下替換依賴來源
5. 是否缺少 project-level 抽象服務層
6. 是否把 project shell 應承擔的 orchestration 反向塞回模組
7. 是否把 module-local 責任上推到 shared / project 全域層

---

## 4. 合理耦合白名單

以下耦合目前視為合理且可接受，不列為本次改善目標：

1. 同一流程鏈共享 ordering context
2. shell 作為 route / layout orchestration
3. staff dashboard 對聚合 API 的依賴
4. staff shell 對 route 與導航投影的依賴

前提是：

1. 不直接依賴他模組私有 store 內臟
2. 不直接依賴他模組私有 session shape
3. 可描述責任邊界

---

## 5. Phase 1：收斂 `staff-auth` 與 `visitor-stats`

### 5.1 目標

把目前最明確違反模組哲學的結構先收掉。

### 5.2 主要問題

1. `staff-auth` 不像模組，卻以模組身分存在
2. `visitor-stats` 直接依賴 `staff-auth` store

### 5.3 Task

1. 將 `staff-auth` 的責任拆分為：
   - project-level component
   - project-level service
   - project-level store 或 adapter
2. 取消 `staff-auth` 作為 `modules/` 成員的身分
3. 讓 `visitor-stats` 不再直接讀 `dineCoreStaffAuthStore`
4. 建立單一 staff permission / token 取得入口
5. 確保 `visitor-stats` 改為透過該入口取得 `isSuperAdmin` 與 token

### 5.4 完成定義

1. `staff-auth` 不再以 module 名義存在
2. `visitor-stats` 不再直接讀 `dineCoreStaffAuthStore`
3. staff token / permission 統一由單一 service 提供
4. `visitor-stats` 的 route/page/API 不再知道 `staff-auth` 的具體 session shape

### 5.5 驗證

1. `tako` 登入後仍可進入每日 IP 訪客統計頁
2. 非 `tako` 帳號仍不可查看每日 IP 訪客統計
3. `visitor-stats` 查詢 API 仍能帶正確 token
4. 專案內不再存在對 `dineCoreStaffAuthStore` 的直接跨模組依賴

---

## 6. Phase 2：收斂顧客流程共享狀態

### 6.1 目標

降低 `entry / menu / cart / checkout / order-tracker` 之間互抓 store 的情況。

### 6.2 主要問題

1. 顧客流程模組直接知道彼此 store 名稱
2. 共享流程狀態沒有 project-level flow abstraction

### 6.3 Task

1. 建立 project-level `orderingFlowService` 或等價 abstraction
2. 將 ordering context、共享流程資料、必要的跨頁流程狀態收斂到該 abstraction
3. 讓 `menu`、`cart`、`checkout`、`order-tracker` 改透過 flow service 工作
4. 逐步取消跨模組直接抓 `entryStore` / `cartStore`

邊界限制：

- `orderingFlowService` 不得退化成新的共享垃圾桶
- 它只承接流程共享上下文
- 不得吸收 page-local UI state

### 6.4 完成定義

1. `entry/menu/cart/checkout/order-tracker` 不再直接互抓彼此 store
2. 共享流程狀態改由 project-level flow service 提供
3. 模組之間若需共享狀態，只能透過明確 flow abstraction 或 API truth

### 6.5 驗證

1. 顧客從入桌到送單的主流程仍可走通
2. 同一 ordering context 仍可跨頁延續
3. 各模組移除直接 `world.store('其他模組Store')` 後仍可運作

---

## 7. Phase 3：整理 `LayoutRoot`

### 7.1 目標

讓 `LayoutRoot` 從耦合中心退回 shell。

### 7.2 主要問題

1. `LayoutRoot` 直接碰太多模組 store
2. `LayoutRoot` 承擔了過多 auth / session / flow 細節

### 7.3 Task

1. 把登入 UI、staff 跳轉規則、權限判斷拆出 `LayoutRoot`
2. 保留 `LayoutRoot` 只負責：
   - layout shell
   - route orchestration
   - 已註冊導航投影
3. 將 shell 對模組 store 的直接依賴降到最低

依賴原則：

- `LayoutRoot` 可以依賴 project-level facade / orchestration service
- 不可直接依賴 module store 名稱與 session shape

### 7.4 完成定義

1. `LayoutRoot` 不再直接呼叫 `login` / `logout` / `loadSession`
2. shell 僅保留 layout / route orchestration 責任
3. shell 不再直接依賴 staff session 的具體實作細節

### 7.5 驗證

1. staff shell 仍正常顯示
2. guest shell 仍正常顯示
3. 移除非核心模組時，shell 可降級，不直接崩潰

---

## 8. Phase Gate

每完成一個 Phase，都必須重新執行一次模組哲學健檢。

### 8.1 Gate 規則

1. 若該 Phase 預期應消除的偏離仍存在，禁止進入下一階段
2. 若出現新的高嚴重度偏離，禁止進入下一階段
3. 若只剩合理耦合白名單內的耦合，可視為通過

### 8.2 Gate 產出

每一階段結束時，至少要輸出以下內容：

1. 本階段原目標偏離是否已消除
2. 是否出現新的偏離
3. 哪些耦合屬於合理耦合
4. 是否允許進入下一階段

固定輸出格式：

1. 本階段目標偏離是否消除
2. 新偏離清單
3. 合理耦合清單
4. Gate 判定：`Pass` / `Blocked`
5. 若為 `Blocked`，必修正項目

---

## 9. 禁止事項

本執行書進行期間，不得：

1. 把重構目標擴張成全面翻新專案
2. 把所有耦合都視為必須消滅
3. 因為功能可運行就跳過結構偏離檢查
4. 在沒有完成當前 Phase Gate 前進入下一階段

---

## 10. 預期可改善程度

若依本執行書完成三個階段，預期可改善到：

1. 明確假模組移出 `modules/`
2. staff 功能不再直接打穿彼此 store
3. 顧客流程模組之間不再直接依賴彼此具體內部狀態
4. `LayoutRoot` 回到 project shell 應有責任

本次改善不追求：

1. 零耦合
2. 每個模組完全彼此無知
3. shell 完全不參與流程編排

本次追求的是：

1. 合法耦合
2. 可理解耦合
3. 可替換邊界
4. 不直接依賴他模組內臟

---

## 11. 執行順序

固定順序如下，不得跳階：

1. 先完成 Phase 1
2. 通過 Phase 1 Gate
3. 再進入 Phase 2
4. 通過 Phase 2 Gate
5. 最後進入 Phase 3
