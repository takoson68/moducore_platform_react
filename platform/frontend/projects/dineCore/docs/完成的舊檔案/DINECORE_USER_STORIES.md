# DineCore User Stories

## 0. 文件用途
本文件將 `dineCore系統規格書.md` 收斂為可開發的 user stories。

本文件用途為：
- 統整 DineCore 的產品需求
- 將需求拆為可實作的故事單位
- 為後續模組切分、資料模型與開發排程提供依據

本文件不負責：
- UI 細節設計
- API 格式定稿
- 資料表結構定稿

---

## 1. 已確認前提
- 掃碼入口採固定桌號模式
- 顧客端全程匿名，不要求登入
- 同桌多人點餐採每人一個子購物車，再於結帳時合單
- 分帳僅支援按人，不支援按品項拆帳
- 第一階段付款採櫃台人工確認，不串接線上支付
- 廚房畫面先不做工作站拆單
- 菜單品項需支援客製選項與加料機制
- 關帳後修正帳務需由店長解鎖

---

## 2. MVP 範圍
MVP 先聚焦以下能力：
- 固定桌號掃碼進入
- 菜單瀏覽與品項客製
- 子購物車與同桌合單
- 櫃台人工付款確認
- 訂單成立與狀態追蹤
- 櫃台訂單管理
- 廚房單一看板
- 基礎營運統計與關帳前檢視

---

## 3. Epic 與 User Stories

### Epic A. 顧客進入與桌號識別

#### US-A01 掃碼進入點餐頁
身為顧客，我希望掃描桌上 QR Code 後可直接進入點餐頁，讓我不需下載 App 或註冊即可開始點餐。

Acceptance Criteria:
- 掃碼後可直接進入對應桌號的點餐入口
- 進入頁面時系統已帶入固定桌號
- 若桌號不存在或停用，系統顯示明確錯誤提示

Priority: MVP

#### US-A02 顯示桌號與用餐情境
身為顧客，我希望進入點餐頁後能清楚看到目前桌號與用餐模式，讓我確認自己在正確桌次中點餐。

Acceptance Criteria:
- 點餐頁顯示當前桌號
- 若為內用模式，桌號資訊不可被顧客任意改寫
- 系統可根據入口設定顯示內用、外帶或自取模式

Priority: MVP

---

### Epic B. 菜單瀏覽與品項客製

#### US-B01 瀏覽分類菜單
身為顧客，我希望菜單依分類清楚呈現，讓我快速找到想點的品項。

Acceptance Criteria:
- 菜單至少支援分類切換
- 每個品項顯示名稱、價格與基本說明
- 分類切換不應導致購物車內容遺失

Priority: MVP

#### US-B02 查看完整品項資訊
身為顧客，我希望看到圖片、價格、售完狀態與成分標示，讓我做出正確選擇。

Acceptance Criteria:
- 品項顯示圖片、名稱、價格
- 品項顯示辣度、肉類、素食、過敏原等標示
- 售完品項需顯示不可下單狀態

Priority: MVP

#### US-B03 客製品項選項
身為顧客，我希望餐點可選擇加麵、加大、加辣、加蔥等客製項目，讓餐點符合需求。

Acceptance Criteria:
- 品項可設定多種客製選項
- 客製選項可影響價格或備註內容
- 顧客在加入購物車前可完成選擇

Priority: MVP

#### US-B04 顯示停售與缺貨
身為顧客，我希望當品項停售或缺貨時能立即看到，避免誤點。

Acceptance Criteria:
- 停售或缺貨品項不可加入購物車
- 畫面需明確標示停售或缺貨狀態
- 櫃台或廚房更新缺貨後，前台可同步反映

Priority: MVP

---

### Epic C. 子購物車、合單與送單

#### US-C01 建立個人子購物車
身為同桌顧客，我希望每個人都能建立自己的子購物車，讓後續按人分帳更清楚。

Acceptance Criteria:
- 同桌可存在多個子購物車
- 每個子購物車可獨立新增、修改與刪除品項
- 子購物車需有可辨識的使用者標記，例如暱稱或序號

Priority: MVP

#### US-C02 管理購物車內容
身為顧客，我希望能修改數量、刪除品項與新增備註，讓購物內容保持正確。

Acceptance Criteria:
- 已加入品項可修改數量
- 已加入品項可刪除
- 每筆購物車品項可輸入備註

Priority: MVP

#### US-C03 同桌合單送出
身為同桌顧客，我希望能將多個子購物車合併為同桌訂單，讓櫃台與廚房只需處理一張單。

Acceptance Criteria:
- 系統可將多個子購物車合併為同一張訂單
- 合單後仍保留每位顧客的點餐歸屬
- 訂單明細可區分不同子購物車來源

Priority: MVP

#### US-C04 防止重複送單
身為顧客，我希望送單時系統能避免重複提交，避免重複下單。

Acceptance Criteria:
- 送單處理中按鈕需鎖定或顯示處理狀態
- 網路延遲時不得產生重複訂單
- 若送單失敗，系統需顯示明確錯誤訊息

Priority: MVP

---

### Epic D. 結帳與人工付款

#### US-D01 顯示透明結帳明細
身為顧客，我希望結帳頁清楚顯示完整明細、總金額、服務費與稅金，讓我確認帳單內容。

Acceptance Criteria:
- 結帳頁顯示品項、單價、數量、小計
- 顯示服務費、稅金與總金額
- 顯示每個子購物車或點餐人的金額歸屬

Priority: MVP

#### US-D02 按人分帳
身為同桌顧客，我希望系統以按人分帳為基礎，讓每個人對自己的帳單負責。

Acceptance Criteria:
- 系統可依子購物車區分每位顧客應付金額
- 不提供按品項拆帳功能
- 若需分帳，必須以不同子購物車建立

Priority: MVP

#### US-D03 櫃台人工確認付款
身為櫃台人員，我希望在顧客到櫃台付款後手動確認付款狀態，讓第一階段先不依賴線上支付串接。

Acceptance Criteria:
- 訂單可標記為未付款、付款確認中、已付款
- 櫃台可依付款方式完成人工確認
- 付款確認後，訂單狀態同步更新

Priority: MVP

#### US-D04 支援現金與現場支付方式
身為櫃台人員，我希望能記錄現金與其他現場支付方式，讓日後報表可對帳。

Acceptance Criteria:
- 至少可記錄現金與手動標記的非現金付款方式
- 每張訂單需保存付款方式
- 報表可依付款方式彙整

Priority: MVP

---

### Epic E. 訂單狀態與顧客追蹤

#### US-E01 顯示下單成功資訊
身為顧客，我希望下單後立即看到成功畫面，包含訂單編號與目前狀態，讓我知道訂單已成立。

Acceptance Criteria:
- 送單成功後顯示訂單編號
- 顯示目前訂單狀態
- 顯示所屬桌號

Priority: MVP

#### US-E02 顧客查看訂單進度
身為顧客，我希望可查看訂單目前進度，降低等待焦慮。

Acceptance Criteria:
- 顧客可看到待處理、製作中、完成、已取餐等狀態
- 狀態變更後前台能更新
- 若店家提供預估等待時間，畫面可顯示

Priority: MVP

---

### Epic F. 櫃台訂單管理

#### US-F01 查看即時訂單列表
身為櫃台人員，我希望看到即時訂單列表並自動排序，避免漏單。

Acceptance Criteria:
- 新訂單進入列表時可清楚辨識
- 列表可依建立時間或狀態排序
- 訂單卡片至少顯示桌號、訂單編號、付款狀態與訂單狀態

Priority: MVP

#### US-F02 搜尋訂單
身為櫃台人員，我希望能依桌號、訂單編號或電話搜尋訂單，加快查詢速度。

Acceptance Criteria:
- 可依桌號搜尋
- 可依訂單編號搜尋
- 若未來有電話資料，也保留搜尋欄位擴充性

Priority: MVP

#### US-F03 管理訂單狀態
身為櫃台人員，我希望更新訂單狀態，讓前後場同步。

Acceptance Criteria:
- 櫃台可切換待處理、製作中、完成、已取餐
- 狀態變更需留下操作紀錄
- 顧客端與廚房端同步看到更新後狀態

Priority: MVP

#### US-F04 改單與取消單
身為櫃台人員，我希望在允許條件下可修改或取消訂單，處理現場例外。

Acceptance Criteria:
- 尚未進入不可修改階段的訂單可調整
- 取消單需填寫原因
- 改單與取消單需記錄操作人與時間

Priority: Phase 2

#### US-F05 補印與異常提示
身為櫃台人員，我希望可補印小白單並收到異常提示，降低作業風險。

Acceptance Criteria:
- 訂單可執行補印或重印
- 系統可提示重複下單或付款異常
- 異常事件需保留紀錄

Priority: Phase 2

---

### Epic G. 廚房看板

#### US-G01 廚房查看未完成訂單
身為廚房人員，我希望只看到未完成訂單，專注處理出餐。

Acceptance Criteria:
- 廚房畫面只顯示未完成訂單
- 每筆訂單顯示桌號、品項、數量與備註
- 可依等待時間排序

Priority: MVP

#### US-G02 完成出餐並同步狀態
身為廚房人員，我希望標示完成後能同步通知櫃台與顧客，讓取餐流程順暢。

Acceptance Criteria:
- 廚房可將訂單或品項標示為完成
- 完成後櫃台與顧客端同步更新
- 系統保留完成時間紀錄

Priority: MVP

#### US-G03 即時設定缺貨
身為廚房或店長，我希望可即時設定缺貨，避免顧客繼續點不到的品項。

Acceptance Criteria:
- 缺貨狀態可從後台更新
- 更新後前台與櫃台立即反映
- 缺貨品項不可再加入新購物車

Priority: MVP

---

### Epic H. 菜單與店務管理

#### US-H01 管理菜單內容
身為店長，我希望能修改菜單價格、售完狀態與隱藏狀態，讓菜單可即時維護。

Acceptance Criteria:
- 可修改價格
- 可設定售完或隱藏
- 更新後前台菜單同步顯示

Priority: Phase 2

#### US-H02 管理桌號與區域
身為店長，我希望可設定桌號與區域，讓內用場景更容易管理。

Acceptance Criteria:
- 可新增、修改、停用桌號
- 桌號可歸屬區域
- 停用桌號不可再作為有效入口

Priority: Phase 2

#### US-H03 設定營業時間與暫停接單
身為店長，我希望能設定營業時間與暫停接單，讓系統符合實際營運狀態。

Acceptance Criteria:
- 可設定營業中與非營業中狀態
- 可手動暫停接單
- 顧客端需看到目前不可下單原因

Priority: Phase 2

#### US-H04 角色權限管理
身為店長，我希望櫃台、廚房與店長擁有不同權限，避免誤操作與權限流動。

Acceptance Criteria:
- 至少區分櫃台、廚房、店長角色
- 不同角色只能看到授權功能
- 高風險操作需限制於店長

Priority: Phase 2

---

### Epic I. 報表、稽核與關帳

#### US-I01 查看基本營運統計
身為店長，我希望查看今日營收、訂單數與熱門品項，讓我掌握當日經營狀況。

Acceptance Criteria:
- 顯示今日營收
- 顯示今日訂單數
- 顯示熱門品項排行

Priority: MVP

#### US-I02 日報表彙整
身為店長，我希望能依日期、時段與付款方式查看日報表，讓我進行營運分析。

Acceptance Criteria:
- 日報表可依日期查詢
- 可區分時段
- 可區分付款方式

Priority: Phase 2

#### US-I03 匯出營收與銷售明細
身為店長，我希望能匯出營收報表與銷售明細，讓我做對帳與外部使用。

Acceptance Criteria:
- 可匯出營收報表
- 可匯出銷售明細
- 匯出內容需包含訂單與付款相關欄位

Priority: Phase 2

#### US-I04 記錄員工操作軌跡
身為店長，我希望每位員工的關鍵操作都有紀錄，讓異常可追蹤。

Acceptance Criteria:
- 記錄改單、取消、付款確認、退款、關帳等操作
- 每筆紀錄包含操作人、時間與動作
- 紀錄不可被一般角色任意刪除

Priority: Phase 2

#### US-I05 每日關帳與店長解鎖
身為店長，我希望每日可執行關帳，關帳後一般人不可再修改帳務，若有例外需由店長解鎖。

Acceptance Criteria:
- 系統支援每日關帳
- 關帳後一般角色不可修改當日帳務
- 若需修正，必須由店長執行解鎖
- 解鎖與後續修正需保留紀錄

Priority: Phase 2

---

## 4. 建議後續工作順序
1. 先將 MVP stories 轉成模組清單與頁面地圖
2. 再補資料模型，例如桌號、菜單、品項選項、子購物車、訂單、付款、關帳紀錄
3. 最後才展開 API 與 UI 細節
