# 顧客端整體驗收表

更新日期：2026-03-11

## 1. 目的

本文件用來驗收 `dineCore` 顧客端主流程是否已形成完整閉環。

本次驗收範圍：

1. `entry`
2. `menu`
3. `cart`
4. `checkout`
5. `order-tracker`

本文件不處理 staff 後台驗收，staff 後台請另依整體收尾清單執行。

## 2. 驗收前提

開始前請先確認：

1. 已有可用桌號，例如 `A01`
2. `menu-admin` 已有至少一筆可販售商品
3. 至少一筆商品帶有客製規則
4. 後端 API 與資料庫可正常連線
5. 若使用真實模式，前端目前為 `apiMode=real`

## 3. 驗收主軸

本次驗收分成五條主軸：

1. 入口與桌號
2. 菜單與客製規則
3. 購物車與金額
4. 結帳與送單
5. 訂單追蹤與續點

## 4. 驗收項目

### 4.1 入口與桌號

#### Case 01：掃桌號入口

步驟：

1. 開啟桌號入口，例如 `/t/A01`
2. 確認頁面可正常載入
3. 確認頁面顯示對應桌號

預期結果：

1. 可正常進入 `entry` 頁
2. 桌號顯示為 `A01`
3. 未出現初始化錯誤

#### Case 02：從 entry 進入菜單

步驟：

1. 在 `entry` 頁點擊進入菜單

預期結果：

1. 跳轉至 `/t/A01/menu`
2. session 不會丟失
3. 顧客端上下文保持一致

### 4.2 菜單與客製規則

#### Case 03：菜單顯示商品與分類

步驟：

1. 進入 `/t/A01/menu`
2. 檢查分類區塊
3. 檢查商品列表

預期結果：

1. 可顯示 `menu-admin` 已設定的分類
2. 可顯示未隱藏、未禁售的商品
3. 商品名稱、描述、價格與後台一致

#### Case 04：客製群組顯示

步驟：

1. 點開帶客製規則的商品
2. 檢查 option group 與 options

預期結果：

1. group 名稱正確
2. `single / multi / required` 顯示正確
3. option 名稱與加價正確
4. default options 顯示正確

#### Case 05：single group 行為

步驟：

1. 在 single group 中切換選項

預期結果：

1. 同群組同時間最多只能選一個
2. 原本選中的項目會被正確替換

#### Case 06：multi group 行為

步驟：

1. 在 multi group 中選取多個選項

預期結果：

1. 可同時選取多個
2. 顯示與計價正確

#### Case 07：required group 行為

步驟：

1. 嘗試不選 required group 就加入購物車

預期結果：

1. 系統會阻止加入
2. 使用者可理解缺少必要選項

### 4.3 購物車與金額

#### Case 08：加入購物車

步驟：

1. 選擇商品
2. 套用客製規則
3. 加入購物車

預期結果：

1. 購物車新增成功
2. 商品名稱與客製內容正確顯示
3. 小計正確

#### Case 09：加價計算

步驟：

1. 選擇含加價 option 的商品
2. 加入購物車

預期結果：

1. 商品價格 = base price + option priceDelta
2. 購物車金額正確

#### Case 10：修改購物車內容

步驟：

1. 在購物車調整商品數量或刪除商品

預期結果：

1. 數量變更正確
2. 刪除後資料正確消失
3. 總金額同步更新

### 4.4 結帳與送單

#### Case 11：進入結帳頁

步驟：

1. 從購物車進入結帳頁

預期結果：

1. 可正常進入 `/t/A01/checkout`
2. 購物車內容完整帶入
3. 金額正確

#### Case 12：送出訂單

步驟：

1. 在結帳頁送出訂單

預期結果：

1. 成功建立訂單
2. 成功跳轉到成功頁或 tracker
3. 櫃台與廚房可看到該筆訂單

#### Case 13：訂單資料一致性

步驟：

1. 送出含客製規則的商品
2. 到 staff 端確認訂單內容

預期結果：

1. 商品名稱正確
2. 客製規則正確
3. 金額正確

### 4.5 訂單追蹤與續點

#### Case 14：進入訂單追蹤

步驟：

1. 送單後進入 `/t/A01/order/{orderId}`

預期結果：

1. 訂單編號正確
2. 訂單狀態正確
3. 顯示內容與送出時一致

#### Case 15：狀態更新同步

步驟：

1. 由 staff 端更新訂單狀態
2. 回顧客端檢查 tracker

預期結果：

1. 顧客端可看到最新狀態
2. 不會出現錯誤狀態跳躍

#### Case 16：續點流程

步驟：

1. 完成一筆訂單後再回到菜單續點

預期結果：

1. 可繼續下新批次
2. 既有訂單追蹤不會被覆蓋
3. session 與桌號不混亂

## 5. 必查風險點

驗收時請特別注意以下風險：

1. `entry` 與 `menu` 之間 session 丟失
2. `default options` 前台顯示正確但送單錯誤
3. 加價在購物車正確、結帳卻錯誤
4. tracker 與 staff 端訂單狀態不同步
5. 同桌續點時覆蓋既有資料

## 6. 驗收結果記錄格式

每次驗收建議至少記錄：

1. 驗收日期
2. 驗收環境
3. 桌號
4. 驗收商品
5. 是否使用真實 API
6. 成功 / 失敗
7. 問題描述

## 7. 驗收完成標準

可視為顧客端主流程驗收完成的條件：

1. 入口、菜單、購物車、結帳、追蹤全流程都可完成
2. 客製規則在顯示、送單、訂單內容三處一致
3. 加價與總金額全流程一致
4. staff 端與 guest 端看到的訂單內容一致
5. 未出現 session 錯亂、桌號錯亂或續點覆蓋問題
