# dineCore 購物車訂單流程執行書

## 0. 文件目的
- 本文件以「瀏覽器暫存購物車，送單才落 DB」為唯一實作方向。
- 目標是把目前的規劃文件轉成可實作、可驗收、可追蹤進度的執行書。
- 本文件預設不做多人共享草稿購物車。
- 本文件預設保留後續按鈕式併單，但不將併單納入第一階段送單閉環。

## 1. 目標結果
- 顧客未送單前，購物車只存在瀏覽器本機。
- 顧客送單時，`checkout-submit` 成為唯一正式落庫入口。
- 後端只保存正式訂單、批次、明細、歷程。
- 舊草稿 cart API 停用。
- `dinecore_cart_items` 明確成為正式送單明細表。
- `dinecore_table_sessions` 第一階段採 B1 弱化保留。

## 2. 範圍

## 3. 核心規則

### 3.1 本機身份
- 進桌號時仍保留 `A01-XXX` 類型的本機身份標籤。
- `entry-context` 仍需回：
  - `orderingSessionToken`
  - `orderingLabel`
  - 桌號可點餐狀態
- 此身份用於：
  - 本機 cart key
  - 送單來源識別
  - 送單後 success / tracker 接續

### 3.2 本機 cart
- cart 不進 DB。
- key 建議：
  - `dinecore.cart.{tableCode}.{orderingSessionToken}`
- restore：
  - 同桌同 token 還原
  - 不同桌號不共用
  - token 改變不沿用
- 清空：
  - 送單成功清空
  - 使用者手動清空清空
  - 超過保存期限可清掉

### 3.3 正式送單
- 前端本地計算價格只作顯示。
- 後端重算價格才是正式依據。
- `checkout-submit` 必須單交易完成：
  - 驗證
  - 建立或接回 order
  - 建立 batch
  - 寫入 items
  - 更新 totals
  - 寫 timeline

### 3.4 防重複提交
- 前端 submit 後立即 disable 按鈕。
- 前端每次提交帶 `clientSubmissionId`。
- 後端以 `orderingSessionToken + clientSubmissionId` 做短時間冪等保護。
- 首次成功與重複成功回應格式一致。

### 3.5 再加點
- 第一階段採保守規則。
- 只有在以下條件成立時才接回既有 order：
  - 同一 `table_code`
  - 同一有效 `orderingSessionToken`
  - order 未 `paid`
  - order 未 `cancelled`
  - order 未 `merged`
- 否則開新 order，交由後續併單處理。
