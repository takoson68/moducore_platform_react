# 模組哲學健檢書

更新日期：2026-03-11

## 1. 目的

本文件用來重新檢查 `platform/frontend/projects/dineCore/` 目前是否仍偏離本次模組哲學改善目標。

本次健檢採用以下判定角度：

1. 是否仍存在假模組
2. 是否仍存在跨模組直接讀取他模組 store / session / shape
3. 是否已建立足夠的 project-level abstraction
4. `LayoutRoot` 是否已回到 shell 責任

---

## 2. 本次健檢結論

本次健檢結果：

> **Phase 1 / Phase 2 / Phase 3 目標均已達成，整體判定為 Pass。**

目前 `dineCore` 已從「staff-auth 假模組 + 顧客流程互抓 store + shell 過度承擔 session 細節」的狀態，
收斂為：

1. staff session 改為 project-level service / API / component
2. `visitor-stats` 改走統一 staff 邊界，不再打穿 staff session 實作
3. 顧客流程改由 project-level `dineCoreOrderingFlowService` 協調
4. `LayoutRoot` 改以 facade / orchestration service 工作，不再直接呼叫 staff session 行為

## 4. 固定尺標檢查

### 4.1 是否直接知道他模組 store 名稱

判定：**已大幅收斂，符合本次目標**

說明：

1. `visitor-stats` 已不再直接讀取 `dineCoreStaffAuthStore`
2. `menu / cart / checkout / order-tracker` 不再直接互抓彼此 store 名稱
3. 顧客流程共享狀態已改由 `dineCoreOrderingFlowService.js` 協調

仍存在的 `world.store(...)`：

1. 多數 page 仍讀取自己的 module-local store
2. 這屬於模組內部合法耦合，不屬於跨模組打穿

### 4.2 是否直接知道他模組 session 結構

判定：**Phase 1 已通過**

說明：

1. `visitor-stats` 不再自己讀 staff session token
2. `LayoutRoot` 不再直接依賴 `dineCoreStaffAuthStore.state.session`
3. staff session 現由 `dineCoreStaffAuthService.js` 統一封裝

### 4.3 是否直接知道他模組資料 shape

判定：**Phase 2 已通過**

說明：

1. 顧客流程共享資料已集中到 `dineCoreOrderingFlowService.js`
2. `checkout/store.js` 不再自行抓 `dineCoreCartStore`
3. `order-tracker` 改由 flow service 提供 ordering session token 與讀取流程

### 4.4 是否可在不改他模組內部的前提下替換依賴來源

判定：**已明顯改善**

說明：

1. staff auth 可在 project-level service / API 層替換，不需再動模組
2. 顧客流程共享狀態可在 ordering flow abstraction 內部調整，不必逐個改 page
3. shell 導航與 staff route 權限已集中在 `dineCoreStaffShellService.js`

### 4.5 是否缺少 project-level 抽象服務層

判定：**已補齊**

已存在的 project-level abstraction：

1. `services/dineCoreStaffAuthService.js`
2. `services/dineCoreStaffShellService.js`
3. `services/dineCoreOrderingFlowService.js`

### 4.6 是否把 project shell 應承擔的 orchestration 反向塞回模組

判定：**已收斂**

說明：

1. `LayoutRoot` 仍保有 route / layout orchestration
2. staff nav 與 fallback path 判定改由 shell facade 提供
3. 顧客流程初始化與恢復邏輯改由 ordering flow service 協調

### 4.7 是否把 module-local 責任上推到 shared / project 全域層

判定：**目前未見新的高嚴重度偏離**

說明：

1. staff session 原本就不應是 module-local，搬到 project-level 是正確收斂
2. ordering flow 只承接共享流程上下文，未觀察到明顯退化成共享垃圾桶
3. 各 page 自己的顯示狀態與 module-local store 仍保留在原模組內

## 5. Phase 1 健檢：`staff-auth` 與 `visitor-stats`

### 5.1 現況判定

判定：**目標偏離已消除**

### 5.2 證據

1. `project.config.js` 已不再列出 `staff-auth`
2. `modules/staff-auth/` 已移除
3. `api/staffSessionApi.js` 承接 staff session API 邊界
4. `services/dineCoreStaffAuthService.js` 承接 session / login / logout / permission 狀態
5. `components/StaffAuthPanel.vue` 承接登入 UI
6. `modules/visitor-stats/pages/VisitorStatsPage.vue` 改用 `useDineCoreStaffAuth()`
7. `modules/visitor-stats/api/visitorStatsApi.js` 改用 `staffApiRequest`

### 5.3 判定說明

目前 `staff-auth` 不再以 module 身分存在，而是明確轉為 project-level auth 能力。

`visitor-stats` 也不再直接依賴 staff store 的具體 session shape，因此 Phase 1 的原始高嚴重度偏離已被消除。

### 5.4 Phase 1 Gate

1. 本階段目標偏離是否消除：**是**
2. 新偏離清單：**無高嚴重度新偏離**
3. 合理耦合清單：
   - `visitor-stats` 對 staff API 邊界的依賴
   - shell 對 project-level auth service 的依賴
4. Gate 判定：**Pass**
5. 若為 Blocked，必修正項目：**無**

## 6. Phase 2 健檢：顧客流程共享狀態

### 6.1 現況判定

判定：**目標偏離已消除**

### 6.2 證據

1. `services/dineCoreOrderingFlowService.js` 已存在
2. `modules/menu/pages/MenuPage.vue` 改用 `useDineCoreOrderingFlow()`
3. `modules/cart/pages/CartPage.vue` 改用 `useDineCoreOrderingFlow()`
4. `modules/checkout/pages/CheckoutPage.vue` 改用 `useDineCoreOrderingFlow()`
5. `modules/checkout/pages/CheckoutSuccessPage.vue` 改用 `useDineCoreOrderingFlow()`
6. `modules/order-tracker/pages/OrderTrackerPage.vue` 改用 `useDineCoreOrderingFlow()`
7. `modules/checkout/store.js` 不再直接抓 `dineCoreCartStore`

### 6.3 判定說明

顧客流程共享上下文目前主要由 ordering flow service 協調：

1. entry context 初始化
2. cart 與 entry 的同步
3. menu item schema 注入
4. checkout 載入與送單後狀態回寫
5. tracker 取得 ordering session token

這代表 `menu / cart / checkout / order-tracker` 之間原本的跨模組直接耦合已明顯下降。

### 6.4 風險觀察

目前 `dineCoreOrderingFlowService.js` 的責任仍屬合理，但後續新增功能時要持續守住邊界：

1. 只承接共享流程上下文
2. 不吸收 page-local UI state
3. 不變成所有顧客端邏輯都往裡塞的共享垃圾桶

### 6.5 Phase 2 Gate

1. 本階段目標偏離是否消除：**是**
2. 新偏離清單：**無高嚴重度新偏離**
3. 合理耦合清單：
   - 顧客流程共享 ordering context
   - flow service 對 entry / cart / menu / checkout / tracker 的協調
4. Gate 判定：**Pass**
5. 若為 Blocked，必修正項目：**無**

## 11. 總結

`dineCore` 目前已從原本最明確的三個問題收斂完成：

1. `staff-auth` 已從假模組改為 project-level 能力
2. `visitor-stats` 已不再打穿 staff session 邊界
3. 顧客流程共享狀態已收斂到 project-level flow abstraction
4. `LayoutRoot` 已回到 shell 應有責任

後續若再演化，建議維持同一原則：

> **project-level 問題放在 project-level abstraction，  
> module 只處理自己的 page / store / service，  
> 不直接依賴他模組內臟。**
