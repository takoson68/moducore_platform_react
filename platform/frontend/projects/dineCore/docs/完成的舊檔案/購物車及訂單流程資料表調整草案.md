# 購物車及訂單流程資料表調整草案

## 0. 文件用途

本文件用於定義 DineCore「主單 + 多批次送單」模型的最小資料表調整方向。

目的不是一次寫完最終 migration，而是先把資料層責任切清楚，避免後續 API 與前端在錯的資料模型上持續堆功能。

---

## 1. 設計目標

本次資料調整要解決以下問題：

- 同一張桌的第一次送單與後續加點無法分批表示
- 已送單內容仍和可編輯購物車混在同一層
- 後端缺少「本批次已鎖定」的結構性表達
- 廚房 / 櫃台 / tracker 難以區分首單與加點單

---

## 2. 目標模型

資料層分成三層：

1. `dinecore_orders`
   - 主單容器
   - 代表這張桌目前這一輪用餐的大單

2. `dinecore_order_batches`
   - 批次送單
   - 每次按下「合併送單」會產生或封存一筆 batch

3. `dinecore_cart_items`
   - 草稿或已送出批次中的實際品項
   - item 必須明確屬於某一個 batch

---

## 3. 現況問題

目前結構中：

- `dinecore_orders`
  - 同時扮演主單、草稿單、已送單狀態容器

- `dinecore_cart_items`
  - 只掛 `order_id`
  - 沒有 `batch_id`

- `dinecore_guest_sessions`
  - 能辨識顧客
  - 但不能區分顧客操作的是哪一個送單批次

這會導致：

- 已送出的內容與未送出的內容共用同一個 item 集合
- checkout submit 後難以做到真正鎖定
- 後續加點只能繼續污染同一張草稿資料

---

## 4. 資料表調整草案

### 4.1 `dinecore_orders`

建議定位：

- 只作為主單容器
- 不再直接承載「這次草稿送單明細」

建議欄位保留：

- `id`
- `order_no`
- `table_code`
- `order_status`
- `payment_status`
- `payment_method`
- `estimated_wait_minutes`
- `subtotal_amount`
- `service_fee_amount`
- `tax_amount`
- `total_amount`
- `created_at`
- `updated_at`

建議語意調整：

- `order_status`
  - 改為主單層狀態，例如：
    - `open`
    - `closed`
    - `cancelled`

說明：

- 若暫時不改欄位名稱，也至少應把語意轉成主單層，而非單次送單狀態

---

### 4.2 新增 `dinecore_order_batches`

用途：

- 表示某一張主單下的每一次送單批次

建議欄位：

- `id`
- `order_id`
- `batch_no`
- `status`
- `submitted_at`
- `locked_at`
- `source_session_token`
- `created_at`
- `updated_at`

建議狀態：

- `draft`
- `submitted`
- `preparing`
- `ready`
- `served`
- `cancelled`

欄位語意：

- `order_id`
  - 所屬主單

- `batch_no`
  - 第幾次送單
  - 第一筆為 `1`
  - 第二次加點為 `2`

- `status`
  - 目前這個批次的生命週期

- `submitted_at`
  - 使用者按下合併送單的時間

- `locked_at`
  - 批次被正式鎖住不可修改的時間

- `source_session_token`
  - 這次按下送單的是哪個匿名顧客 session

索引建議：

- `INDEX idx_batches_order_status (order_id, status)`
- `UNIQUE KEY ux_batches_order_batch_no (order_id, batch_no)`

---

### 4.3 調整 `dinecore_cart_items`

目前問題：

- 只有 `order_id`
- 無法知道 item 屬於哪一次送單批次

建議新增欄位：

- `batch_id INT UNSIGNED NOT NULL`

建議關聯：

- `FOREIGN KEY (batch_id) REFERENCES dinecore_order_batches(id)`

建議保留欄位：

- `order_id`
- `table_code`
- `cart_id`
- `menu_item_id`
- `title`
- `quantity`
- `price`
- `note`
- `options_json`
- `selected_option_ids_json`
- `created_at`
- `updated_at`

說明：

- `order_id` 可保留作查詢便利與歷史相容
- 但真正的歸屬判準應改為 `batch_id`

---

### 4.4 `dinecore_guest_sessions`

本表原則上可保留。

用途仍為：

- 匿名顧客身份
- 每位顧客自己的 `cart_id`
- 同桌共桌點餐的 participant source

本表目前不必新增欄位的前提：

- 只要 API 在 runtime 能把 session 對應到當前 `draft batch`

若後續需要更強的追蹤能力，可考慮新增：

- `current_batch_id`

但這不是第一階段必要項目。

---

## 5. 資料流規則

### 建立桌內主單

- 若桌號目前沒有 `open` 主單
- 建立一張新的 `dinecore_orders`

### 建立第一個 draft batch

- 新主單建立後
- 自動建立 `batch_no = 1`、`status = draft`

### 送單

- 將目前 `draft batch` 改為 `submitted`
- 寫入 `submitted_at` / `locked_at`
- 接著建立下一個新的 `draft batch`

### 加點

- 後續 item 一律寫進最新的 `draft batch`
- 不可寫回上一個已 `submitted` 的 batch

---

## 6. 查詢策略草案

### 購物車頁

來源：

- 只查當前 `draft batch`

回傳：

- 目前顧客自己的 cart
- 同桌所有 active session 數量
- 目前 draft batch 下的 carts
- 已送出批次摘要

### 結帳頁

來源：

- 只查當前 `draft batch`

### 訂單追蹤頁

來源：

- 查主單
- 展開其下全部 `batches`

### 櫃台 / 廚房

來源：

- 批次為主
- 主單摘要為輔

---

## 7. Migration 建議順序

### Phase A：加新表，不破舊流程

- 新增 `dinecore_order_batches`
- `dinecore_cart_items` 新增 `batch_id`，先允許回填

### Phase B：回填舊資料

- 針對既有 `order_id`
- 每張舊單先補一個 `batch_no = 1`
- 將既有 cart items 掛到該 batch

### Phase C：切 API 到新模型

- cart / checkout / tracker 改走 batch

### Phase D：清理舊語意依賴

- 移除仍把 `order` 當草稿單的邏輯

---

## 8. 最小 SQL 草案

以下僅為結構草案，不代表最終可直接執行版本：

```sql
CREATE TABLE dinecore_order_batches (
  id INT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY,
  order_id INT UNSIGNED NOT NULL,
  batch_no INT UNSIGNED NOT NULL,
  status VARCHAR(32) NOT NULL DEFAULT 'draft',
  submitted_at DATETIME NULL,
  locked_at DATETIME NULL,
  source_session_token VARCHAR(128) NOT NULL DEFAULT '',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  CONSTRAINT fk_dinecore_order_batches_order
    FOREIGN KEY (order_id) REFERENCES dinecore_orders(id)
    ON DELETE CASCADE
    ON UPDATE CASCADE,
  UNIQUE KEY ux_dinecore_order_batches_order_batch_no (order_id, batch_no),
  INDEX idx_dinecore_order_batches_order_status (order_id, status)
);

ALTER TABLE dinecore_cart_items
ADD COLUMN batch_id INT UNSIGNED NULL AFTER order_id;
```

## 9. 驗證案例

資料層至少要能支撐以下情境：

1. A 進桌號，建立主單與 `batch-1 draft`
2. B 進桌號，共用同一主單與同一 draft batch
3. A/B 各自加商品到 `batch-1`
4. A 送單，`batch-1` 變 `submitted`
5. 系統建立 `batch-2 draft`
6. B 再加點時，商品進 `batch-2`
7. tracker 可列出 `batch-1` 與 `batch-2`
