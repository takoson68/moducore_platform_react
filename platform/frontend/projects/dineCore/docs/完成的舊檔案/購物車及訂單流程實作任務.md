# 購物車及訂單流程實作任務

## 0. 文件用途

本文件用於定義 DineCore「購物車送單、批次鎖定、後續加點」的最小實作任務清單。

目標不是重新討論方向，而是把已確認的流程拆成可落地的工程步驟。

---

## 1. 目標

將目前「單一草稿訂單反覆共用」的流程，調整為：

- 一張桌對應一張主單
- 主單底下可有多個送單批次
- 每次合併送單後，該批次立即鎖定
- 後續加點必須進入新的草稿批次

---

## 2. 成功條件

完成後必須同時滿足：

- A 顧客送單後，該批次不可再被任何顧客修改
- B 顧客輪詢到新狀態後，舊購物車內容不再可編輯
- B 顧客若再加點，必須進入新的 draft batch
- 訂單追蹤頁可區分第一筆送單與後續加點批次
- 廚房 / 櫃台能看出各批次的獨立狀態

---

## 3. 任務清單

### Task 1：定義主單與批次資料模型

目的：
- 建立 `order` 與 `batch` 的清楚分工

實作：
- 保留 `dinecore_orders` 作為主單容器
- 新增 `dinecore_order_batches`
- 為 `dinecore_cart_items` 新增 `batch_id`

完成標準：
- DB schema 可表達「一張主單下有多個批次」
- cart item 不再只屬於整張 order

---

### Task 2：建立批次狀態機制

目的：
- 明確定義哪一批可編輯、哪一批已鎖定

實作：
- 為 `dinecore_order_batches` 定義至少以下狀態：
  - `draft`
  - `submitted`
  - `preparing`
  - `ready`
  - `served`
  - `cancelled`

完成標準：
- 後端可依 batch status 判斷是否允許修改

---

### Task 3：調整 entry-context 取得目前 draft batch

目的：
- 顧客進桌號後，不是只拿主單，而是拿到目前可編輯的草稿批次

實作：
- `entry-context` 取得或建立主單
- `entry-context` 取得目前最新 `draft batch`
- response 補充：
  - `current_batch_id`
  - `current_batch_no`

完成標準：
- 顧客進頁後已綁定到目前唯一可編輯的批次

---

### Task 4：調整 cart API 只操作 draft batch

目的：
- 避免已送單批次被再次修改

實作：
- `GET /carts`
  - 只回目前 `draft batch` 的可編輯資料
  - 另外回已送出批次摘要
- `POST /cart/add-item`
  - item 寫入目前 `draft batch`
- `POST /cart/update-item`
  - 僅允許修改目前 `draft batch`
- `POST /cart/change-item-quantity`
  - 僅允許修改目前 `draft batch`

完成標準：
- 已送單批次不再被 cart API 修改

---

### Task 5：重寫 checkout-submit 的鎖定流程

目的：
- 每次送單後立即封存該批次，並開啟新批次供後續加點

實作：
- `checkout-submit` 將目前 `draft batch` 改為 `submitted`
- 寫入：
  - `submitted_at`
  - `locked_at`
- 建立新的空 `draft batch`
- response 回傳：
  - `submitted_batch_id`
  - `submitted_batch_no`
  - `next_batch_id`
  - `next_batch_no`

完成標準：
- 送單後同一批資料不可再改
- 系統已自動準備下一個空的加點批次

---

### Task 6：調整前端購物車頁批次切換邏輯

目的：
- A 送單後，B 輪詢時自動脫離舊批次，切到新的 draft batch

實作：
- cart polling 發現目前 batch 已非 `draft` 時：
  - 清掉舊編輯狀態
  - 切到新 draft batch
- UI 顯示：
  - 目前加點購物車
  - 目前共桌顧客數
  - 已送出批次摘要

完成標準：
- B 不會再看到可編輯的舊批次內容

---

### Task 7：調整結帳頁只顯示目前 draft batch

目的：
- 避免已送單批次與待送單批次混在一起

實作：
- checkout summary 僅統計目前 `draft batch`
- 若要顯示歷史批次，必須明確分區且唯讀

完成標準：
- 使用者提交的是當前批次，不是整張桌全部歷史內容

---

### Task 8：調整訂單追蹤頁顯示 batch 列表

目的：
- 讓使用者與店家看得出是第一次送單還是後續加點

實作：
- order tracker 顯示：
  - 主單摘要
  - `batches[]`
  - 每個 batch 的品項、狀態、送單時間

完成標準：
- tracker 可看出批次順序與批次狀態

---

### Task 9：調整櫃台與廚房資料來源

目的：
- 商家端應依批次工作，而不是只看混合後的大單

實作：
- 廚房看板以 batch 為主要單位
- 櫃台頁面同時可看：
  - 主單累計摘要
  - 各批次明細

完成標準：
- 廚房與櫃台能識別每輪加點

---

### Task 10：補 migration / seed / 測試資料

目的：
- 讓本機與新環境可重建這套批次模型

實作：
- 補 `sql/dinecore/` 對應 schema 變更
- 補 demo seed 對應 batch 結構
- 補至少一組：
  - 首次送單
  - 第二輪加點

完成標準：
- 本機可重建批次資料模型
- demo 資料可驗證兩輪以上送單流程

---

5. Task 5：checkout-submit
6. Task 6：購物車頁
7. Task 7：結帳頁

9. Task 9：櫃台與廚房
10. Task 10：migration / seed / 測試資料

---

## 5. 驗證案例

至少驗證以下情境：

1. A 顧客進桌號，建立第一個 draft batch
2. B 顧客進桌號，加入同一張桌的共桌流程
3. A、B 各自加商品
4. A 合併送單
5. B 輪詢後不再能修改上一批
6. B 再加點時，商品進入新 draft batch
7. tracker 可看到 batch-1 與 batch-2
8. 櫃台 / 廚房可分辨兩批資料

---

## 6. 實作注意事項

- 不要讓 `menu / carts / checkout` 自行建立新顧客 session
- 不要讓 cart item 同時混用「主單層」與「批次層」語意
- 不要讓已送單批次仍留在可編輯購物車資料集中
- 不要只修前端畫面而不修後端鎖定邏輯
