# 下一階段藍圖

更新日期：2026-03-03

## 1. 靈感主題
- 桌號管理升級為「空間配置管理」
- 不只管理桌號清單，而是管理店內空間、區域、桌位位置與視覺配置

## 2. 目標想像
- 店家可自訂義店內空間顯示
- 店家可管理不同區域，例如內場、外場、吧台區、包廂區
- 桌位不再只是列表，而是放在平面圖上的可視物件
- 管理者可直接看到桌位放置位置、桌型、狀態與入口對應

## 3. 建議技術方向

### 3.1 資料層
- 使用 `JSON` 定義空間配置資料
- 資料內容可包含：
  - `zones`
  - `tables`
  - `x`
  - `y`
  - `width`
  - `height`
  - `shape`
  - `rotation`
  - `label`
  - `status`

### 3.2 視覺層
- 使用 `SVG` 作為空間配置渲染層
- 適合做：
  - 桌位顯示
  - 區域框選
  - 狀態上色
  - 拖拉移動
  - 縮放預覽

### 3.3 模組定位
- 不建議直接塞進目前 `table-admin`
- 建議未來獨立成：
  - `floor-admin`
  - 或 `layout-admin`

## 4. 為什麼值得做
- 桌號管理會從純列表升級成現場營運工具
- 更適合餐廳實際情境
- 可延伸到桌況管理、保留桌、合併桌、區域停用
- 可作為未來平板現場控台的基礎

## 5. 為什麼現在先不做
- 這已經不是單純 CRUD，而是視覺編輯器
- 需要額外處理拖拉、縮放、版面資料、編輯狀態與儲存格式
- 目前 DineCore 主線仍應先完成基本營運能力
- 若現在插入，會提高整體複雜度並影響主線進度

## 6. 建議時機
- 放入 `2.0`
- 或放入基本功能完成後的後續版本
- 建議在 `reports / audit-close / 正式 API` 之後再評估是否啟動

## 7. 未來若啟動可先做的最小版
1. 先支援區域與桌位座標資料
2. 先用 `SVG` 做靜態配置預覽
3. 再補桌位拖拉編輯
4. 最後才補完整空間編輯器能力

## 8. 結論
- 這個功能可實踐
- 技術方向合理
- 但應視為下一階段或 2.0 藍圖
- 目前先保留靈感，不納入當前主線開發
