# 員工後台下一步執行書

更新日期：2026-03-11

## 1. 結論

本輪原先鎖定的 `dineCore` 員工後台主優先項目是 `menu-admin`。
截至本次更新，`menu-admin` 的三個階段已全部完成：

- `P0` 分類管理真實 API
- `P1` 商品狀態 / 價格 / 分類真實 API
- `P2` 客製規則真實 API

因此，`menu-admin` 已不再是員工後台的主要缺口。
後續員工後台的下一步，應轉向其他 staff 模組的正式啟用與營運閉環。

## 2. 已完成範圍

### 2.1 P0 分類管理

已完成：

1. `create-category`
2. `update-category`
3. `delete-category`
4. `reorder-categories`

達成結果：

- `apiMode=real` 可真實寫入分類資料
- 成功後統一回傳完整 `categories + items` snapshot
- `delete-category` 會阻止刪除仍有商品的分類
- `reorder-categories` 已改為完整排序陣列 + transaction

### 2.2 P1 商品管理

已完成：

1. `update-item-status`
2. `update-item-price`
3. `update-item-category`

達成結果：

- `apiMode=real` 可真實寫入商品狀態、價格、分類
- 成功後回傳單筆 `item`
- 前端沿用現有 item patch 更新模式

### 2.3 P2 客製規則

已完成：

1. `add-option-group`
2. `update-option-group`
3. `delete-option-group`
4. `add-option`
5. `update-option`
6. `delete-option`
7. `update-default-options`

達成結果：

- `apiMode=real` 已不再只走 mock
- 成功後統一回傳更新後單筆 `item`
- 沿用既有欄位：
  - `dinecore_menu_items.option_groups_json`
  - `dinecore_menu_items.default_option_ids_json`
- `single` 群組最多只會保留一個預設值
- 刪除 group / option 後，default options 不會殘留髒資料
- 非法 option id 不會被寫入 default options

既然 `menu-admin` 已完成，員工後台後續優先順序建議改為：

1. `dashboard` 正式啟用
2. `reports` 正式啟用
3. `audit-close` 正式啟用
