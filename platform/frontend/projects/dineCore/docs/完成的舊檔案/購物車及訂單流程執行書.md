# 購物車及訂單流程執行書

## 0. 文件用途

本文件用於作為 DineCore「購物車及訂單流程重構」的正式執行書。

本文件的目標不是補充概念，而是提供一套可以：

- 依序執行
- 每階段自動檢查
- 檢查通過後進入下一步
- 直到整條流程完成

的落地執行方式。

---

## 1. 執行目標

本執行書要完成的最終結果是：

- 同桌點餐改為「主單 + 多批次送單」模型
- 每次送單後，該批次立即鎖定
- 後續加點進入新的 draft batch
- 顧客端輪詢能自動感知批次切換
- tracker / 櫃台 / 廚房都能辨識批次

---

## 2. 執行原則

### Rule 1：每個 Phase 必須有明確完成條件

- 未通過檢查，不得進下一步

### Rule 2：每個 Phase 完成後必須執行驗證

- 驗證失敗，不得宣告完成

### Rule 3：先資料層，後流程層，再頁面層

- 不得先改 UI 再回頭補資料模型

### Rule 4：每個 Phase 只做當階段最小必要改動

- 不在中途混入額外功能

---

## 3. 執行總覽

整體執行分為六個 Phase：

1. Phase A：資料模型建置
2. Phase B：資料回填與相容
3. Phase C：顧客端核心 API 切換
4. Phase D：前端 cart / checkout 切換
5. Phase E：tracker / 商家端切換
6. Phase F：全流程驗證與收尾

---

## 4. Phase A：資料模型建置

### 目標

建立 batch 模型所需的最低資料表結構。

### 執行項目

- 建立 `dinecore_order_batches`
- 為 `dinecore_cart_items` 新增 `batch_id`
- 補齊索引與外鍵

### 必改檔案

- `platform/backend/sql/dinecore/*`

### 自動檢查項目

- `dinecore_order_batches` 存在
- `dinecore_cart_items.batch_id` 存在
- `order_id + batch_no` 唯一鍵存在
- `batch_id` 外鍵存在

### 完成條件

- schema 可建立成功
- 沒有外鍵錯誤

### 通過後才可進入

- Phase B

---

## 5. Phase B：資料回填與相容

### 目標

讓現有資料可以被新 batch 模型接受，不因缺欄位而中斷。

### 執行項目

- 每張既有主單補一筆 `batch_no = 1`
- 將現有 `dinecore_cart_items` 掛到對應 batch
- 補 seed / demo data 相容格式

### 必改檔案

- `platform/backend/sql/dinecore/*`

### 自動檢查項目

- 現有 order 是否都有至少一筆 batch
- 現有 cart item 是否都有 `batch_id`
- seed 匯入後是否能成功查到 batch 結構

### 完成條件

- 舊資料仍可查
- 新資料已有 batch 關聯

### 通過後才可進入

- Phase C

---

## 6. Phase C：顧客端核心 API 切換

### 目標

讓後端顧客端 API 改以 batch 為真實流程單位。

### 執行項目

- 調整 `entry-context`
- 調整 `GET /carts`
- 調整 `add-item / update-item / change-item-quantity`
- 重寫 `checkout-submit`
- 調整 `checkout-summary`
- 調整 `order-tracker`

### 必改檔案

- `platform/backend/src/Controllers/DineCoreGuestApiController.php`

### 自動檢查項目

- `entry-context` 可回 `current_batch_id`
- `GET /carts` 可回 `currentBatchId`
- cart mutation 只能寫入 `draft batch`
- `checkout-submit` 後：
  - 舊 batch 變 `submitted`
  - 新 batch 變 `draft`
- `menu / carts / checkout` 不會偷偷新建 session

### 完成條件

- 送單後 DB 狀態正確切換
- 舊批次不可改
- 新批次可用

### 通過後才可進入

- Phase D

---

## 7. Phase D：前端 cart / checkout 切換

### 目標

讓前端顧客頁面正確承接 batch 切換，不殘留舊草稿語意。

### 執行項目

- entry store 補 `currentBatch*`
- cart store 補 `currentBatch* / participantCount / submittedBatches`
- cart page 輪詢切換 batch
- checkout page 失效處理
- menu page 對 `BATCH_LOCKED` 做重抓上下文

### 必改檔案

- `platform/frontend/projects/dineCore/modules/entry/*`
- `platform/frontend/projects/dineCore/modules/menu/*`
- `platform/frontend/projects/dineCore/modules/cart/*`
- `platform/frontend/projects/dineCore/modules/checkout/*`

### 自動檢查項目

- cart page 輪詢後可感知 batch 改變
- A 送單後，B 停留 cart 頁能切到新 draft batch
- checkout page 不可重送舊批次
- menu add-item 不可加進舊批次

### 完成條件

- 前端所有顧客端頁面已改以 batch 為主

### 通過後才可進入

- Phase E

---

## 8. Phase E：tracker / 商家端切換

### 目標

讓批次語意往下傳到 tracker、櫃台與廚房。

### 執行項目

- tracker 顯示 `batches[]`
- 櫃台可看主單總覽與批次明細
- 廚房以 batch 為主要單位

### 必改檔案

- `platform/frontend/projects/dineCore/modules/order-tracker/*`
- `platform/frontend/projects/dineCore/modules/counter/*`
- `platform/frontend/projects/dineCore/modules/kitchen/*`
- `platform/backend/src/Controllers/DineCoreStaffApiController.php`

### 自動檢查項目

- tracker 能顯示批次 1 / 2 / 3
- 櫃台能識別首單與加點單
- 廚房收到的是批次單，不是混合單

### 完成條件

- 商家端與顧客端語意一致

### 通過後才可進入

- Phase F

---

## 9. Phase F：全流程驗證與收尾

### 目標

驗證整條流程已從資料層到頁面層打通。

### 執行項目

- 補 demo seed
- 執行雙瀏覽器驗證
- 執行送單後輪詢驗證
- 執行加點第二批驗證
- 執行 tracker / 櫃台 / 廚房驗證

### 自動檢查項目

- A / B 同桌進入
- A / B 各自加商品
- A 送單
- B 輪詢自動切到新 draft
- B 再加點進入 batch-2
- tracker 能同時看到 batch-1 / batch-2
- 櫃台 / 廚房能分開識別兩批

### 完成條件

- 全流程通過
- 不再存在「送單後舊購物車仍可被修改」問題

---

## 10. 執行節點控制

### Gate A

若 Phase A 或 B 未完成：

- 禁止進入 controller 重寫

### Gate B

若 Phase C 未完成：

- 禁止進入前端 cart / checkout 邏輯切換

### Gate C

若 Phase D 未完成：

- 禁止宣告顧客端流程完成

### Gate D

若 Phase E 未完成：

- 禁止宣告整體訂單流程完成

---

## 11. 建議每階段的執行輸出

每完成一個 Phase，至少回報：

- 改了哪些檔案
- 通過了哪些檢查
- 還剩什麼風險
- 下一步是哪一個 Phase

---

## 12. 建議自動檢查清單

以下項目可作為每次 Phase 完成後的最低檢查：

### DB 檢查

- batch table 是否存在
- batch 狀態是否正確切換
- item 是否掛到正確 batch

### API 檢查

- `entry-context`
- `carts`
- `checkout-summary`
- `checkout-submit`
- `order-tracker`

### 前端檢查

- build 是否成功
- cart polling 是否可運作
- 送單後頁面是否切換

---

## 13. 最小一次做到完成的執行順序

若目標是「不要拆太碎，一次推進到可用」，建議照這個順序：

1. Phase A
2. Phase B
3. Phase C
4. 立即做最小 API 驗證
5. Phase D
6. 立即做雙瀏覽器驗證
7. Phase E
8. Phase F

---

## 14. 實作啟動建議

若下一步要正式開始寫 code，建議從：

- Phase A：資料模型建置

原因：

- 這是整條流程的基座
- 不先改資料層，後面 API 與頁面都會反覆重寫
