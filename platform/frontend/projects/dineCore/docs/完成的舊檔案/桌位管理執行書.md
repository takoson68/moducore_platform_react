# 桌位管理執行書（QR 產生器）

更新日期：2026-03-06  
目標：在「桌位管理」頁面新增「產生 QR」功能，將桌號網址 QR 圖片實際產出並存放於公開路徑 `assets/QRC/`。

## 0. 需求摘要
1. 現況只有「複製連結、下載 QR」但沒有穩定存在的 QR 圖片檔。
2. 新增一個「產生 QR」按鈕。
3. 按下後由後端產生 QR PNG，儲存到公開資料夾 `assets/QRC/`。
4. 桌位管理可直接顯示/管理該 QR 圖片，功能完整閉環。

---

## 1. 路徑與命名規則
1. 公開儲存路徑：
   - `platform/backend/public/assets/QRC/`
2. 檔名規則（固定且可覆蓋）：
   - `{table_code}.png`
   - 例如：`A01.png`
3. 公開 URL：
   - `/assets/QRC/{table_code}.png`
4. URL 內容（QR 內嵌）：
   - 必須使用完整網址（含協定與網域），例如：`https://your-domain.com/t/{table_code}`
   - 不可只寫相對路徑 `/t/{table_code}`（手機掃碼器常無法正確開啟）
   - 來源：由後端設定 `APP_BASE_URL`（或等價設定）組合桌號路徑

完成判定：
- 指定桌號可對應到固定檔案與固定公開網址。

---

## 2. 後端實作步驟

### 2.0 依賴管理前置（先做）
1. 確認 backend PHP 依賴管理方式：
   - 若有 Composer：在 backend 專案安裝 QR 套件並更新 autoload。
   - 若無 Composer：先補齊依賴管理方案，或改採「前端生成 PNG + 後端存檔」替代流程。
2. 明確記錄 QR 產生器來源（套件名稱/版本）。

完成判定：
- 專案可穩定取得「輸入文字 -> 輸出 PNG bytes」能力。

### 2.1 新增 API 路由
1. 在 `platform/backend/src/routes.php` 新增：
   - `POST /api/dinecore/staff/tables/generate-qr`
2. 權限角色：
   - `counter` / `deputy_manager` / `manager`（依你現行規則可調整）

完成判定：
- 路由存在，呼叫不再出現 `NOT_FOUND`。

### 2.2 Controller 新增方法
1. 在 `DineCoreStaffApiController` 新增 `generateTableQr()`。
2. 輸入：
   - `table_code`
3. 驗證：
   - `table_code` 必填
   - 桌號存在於 `dinecore_tables`
4. 產出流程：
   - 建立 QR 內容（桌號入桌 URL）
   - 生成 PNG 二進位
   - 確保 `public/assets/QRC/` 存在（不存在則建立）
   - 寫入 `{table_code}.png`（可覆蓋）
5. 回傳：
   - `ok: true`
   - `data.tableCode`
   - `data.fileName`
   - `data.publicUrl`
   - `data.updatedAt`

完成判定：
- API 回傳成功且磁碟上確實有 PNG 檔案。

### 2.3 QR 產生套件
1. 選定 PHP QR 套件（例如 `endroid/qr-code` 或現有專案已用套件）。
2. 安裝到 backend 專案。
3. 封裝最小依賴寫法，避免 Controller 過重。

完成判定：
- 可透過單一函式輸入文字並輸出 PNG bytes。

## 3. 前端實作步驟（桌位管理頁）

### 3.1 新增 UI 按鈕
1. 在桌位列表每列新增按鈕：`產生 QR`。
2. 點擊後呼叫 `generateTableQr(tableCode)`。
3. 按鈕 loading/disabled 狀態避免重複提交。

完成判定：
- 畫面每筆桌位都有可操作的 `產生 QR`。

### 3.2 串接 API
1. `table-admin/service.js` 新增 real API 呼叫：
   - `POST /api/dinecore/staff/tables/generate-qr`
2. 保留 mock fallback（若目前模組仍支援 mock）。
3. 成功後更新該桌資料：
   - `qrUrl` 設為 `/assets/QRC/{table_code}.png`

完成判定：
- 點按按鈕後，前端可拿到後端回傳的 `publicUrl` 並更新畫面。

### 3.3 顯示與管理
1. 在桌位列顯示：
   - QR 圖片預覽（`img`）
   - 開啟連結（新分頁）
2. 保留既有「複製連結、下載 QR」並改為使用實際檔案 URL。

完成判定：
- 不再是前端臨時生成，實際使用公開檔案。

## 4. 錯誤碼與提示
1. `TABLE_CODE_REQUIRED`
2. `TABLE_NOT_FOUND`
3. `QR_GENERATE_FAILED`
4. `QR_SAVE_FAILED`
5. `STAFF_SESSION_REQUIRED`
6. `STAFF_ROLE_FORBIDDEN`

前端提示文案：
1. 成功：`已產生 {tableCode} 的 QR 圖片`
2. 失敗：顯示錯誤碼與可讀訊息

---

## 5. 驗收步驟（手動）
1. 開啟桌位管理頁，選一個桌號（如 `A01`）。
2. 點擊 `產生 QR`。
3. 確認 API 回傳成功。
4. 檢查檔案存在：
   - `platform/backend/public/assets/QRC/A01.png`
5. 瀏覽器開啟：
   - `/assets/QRC/A01.png`
6. 掃碼測試：
   - 內容導向完整網址（例如 `https://your-domain.com/t/A01`）
7. 重複再產生一次確認可覆蓋更新。

完成判定：
- 檔案可生成、可覆蓋、可掃碼、可公開存取。

---

## 6. 安全與維運注意
1. `table_code` 僅允許白名單字元（英數、`-`、`_`）。
2. 禁止路徑穿越（不可讓使用者組成任意檔名）。
3. 圖片檔案只寫入 `assets/QRC/`。
4. 如日後有大量桌位，建議加上批次重建指令（CLI）。
5. 區分環境網址：
   - `dev` 驗收以 backend host 為主（例：`http://127.0.0.1:8000/assets/QRC/A01.png`）
   - `prod` 驗收以正式網域為主（例：`https://your-domain.com/assets/QRC/A01.png`）

---

## 7. 回滾方案
1. 前端先隱藏 `產生 QR` 按鈕（feature flag）。
2. 後端保留路由但回傳停用訊息，或先移除路由映射。
3. 保留已產生檔案，不做自動刪除。
