# dineCore 收尾清單

更新日期：2026-03-11

## 1. 目前狀態

`dineCore` 目前已完成的主軸如下：

- `menu-admin` 真實 API 已補齊
- `dashboard` 已正式啟用
- `reports` 已正式啟用
- `audit-close` 已正式啟用

所以目前階段已不是補單一 staff 模組的 CRUD，而是進入整體驗收、規則整理與部署收尾。

## 2. 收尾優先順序

建議依照以下順序執行：

1. 顧客端整體驗收
2. 員工後台橫向驗收
3. 關帳規則文件化
4. 部署與編譯流程固定化
5. 驗收清單文件補齊
6. 風險點標記

## 3. 顧客端整體驗收

目標：確認 `entry -> menu -> cart -> checkout -> order-tracker` 在真實資料下完整可用。

驗收重點：

1. 掃桌號入口後可正確進入 `entry`
2. 桌號與 ordering session 可正確建立與延續
3. `menu-admin` 修改過的商品、分類、客製規則會正確顯示在顧客端
4. 加購、刪除、修改客製內容後，購物車資料仍正確
5. 結帳送單成功後，櫃台與廚房都能收到資料
6. `order-tracker` 可正確顯示訂單狀態與續點流程

## 4. 員工後台橫向驗收

目標：確認 staff 模組之間的資料口徑與操作流程一致。

建議驗收順序：

1. `dashboard`
2. `counter`
3. `kitchen`
4. `reports`
5. `audit-close`
6. `menu-admin`
7. `table-admin`

驗收重點：

1. 今日營收只計入已付款訂單
2. `dashboard`、`reports`、`audit-close` 的金額口徑一致
3. 櫃台收款、廚房出單、報表統計的資料不互相矛盾
4. 關帳後櫃台與廚房功能凍結
5. 解鎖後櫃台與廚房功能恢復
6. 菜單更新後，staff 與 guest 端看到的資料一致

## 5. 關帳規則文件化

目標：把目前已確認可用的帳務規則寫清楚，避免日後誤解。

目前已確認規則：

1. 關帳後，櫃台與廚房功能會凍結
2. 解鎖後，櫃台與廚房功能會恢復
3. 目前 `audit-close` 是以 `business_date` 為單位鎖定
4. 目前不是班次制帳務模型，而是日結模型
5. `shift_handover` 目前只是 reason type，不代表真的切出新的帳務區段

這部分建議後續補成獨立文件，避免現場把「交班交接」誤解成「開新帳日」。

## 6. 部署與編譯流程固定化

目標：避免 source code 已修改，但站點仍吃舊 build 產物。

需要固定的流程：

1. 開發完成後執行 `VITE_PROJECT=dineCore npm run build`
2. 明確區分 source code 與部署產物
3. 部署時確認 `platform/backend/public` 是否為最新版本
4. 若目前是 source-only 工作流，要知道本機修改不等於站點已更新

## 7. 驗收清單文件補齊

目標：把驗收流程從口頭記憶改成可重複執行的 checklist。

建議至少補兩份：

1. 顧客端整合驗收表
2. 員工後台整合驗收表

每份至少要列出：

1. 操作步驟
2. 預期結果
3. 失敗時要檢查的模組
4. 是否與真實資料驗證相關

## 8. 風險與待確認事項

目前最值得標記的不是單一 bug，而是模型限制：

1. `audit-close` 目前不是班次帳務系統
2. `shift_handover` 目前仍屬於日結模型內的原因分類
3. 若未來需要同日多次交班結帳，現有資料模型不夠
4. 部署流程若不固定，容易再次出現「改了 source 但站點沒更新」的誤判

## 9. 建議下一步

如果只選一個最值得先做的收尾項目，建議先做：

1. 顧客端整體驗收

原因：

- `menu-admin` 已完成，現在要驗證後台改動是否真的閉環到顧客端
- 這一步完成後，才更能確認 `dineCore` 已進入穩定營運狀態
