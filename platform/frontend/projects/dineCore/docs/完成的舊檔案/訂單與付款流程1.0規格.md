# 訂單與付款流程 1.0 規格（不含帳冊）

## 1. 目標範圍
- 1.0 只做到：顧客送單、櫃台看單、櫃台標記付款。
- 1.0 不做：帳冊建檔、日結分錄、退款沖銷（延到 2.0）。

## 2. 流程定義
1. 顧客送出購物車（checkout submit）
- 產生或更新對應訂單資料。
- 訂單狀態進入 `pending`。
- 付款狀態預設 `unpaid`。

2. 櫃台查看訂單
- 以「訂單」為主體顯示，不以 table session 當主資料來源。
- 顯示：訂單編號、桌號、金額、批次、付款狀態、製作狀態。

3. 櫃台標記付款
- 僅更新訂單付款狀態為 `paid`。
- 同步將該單相關顧客 session 失效（避免續點混單）。
- 不建立帳冊資料（1.0 規格）。

## 3. 資料表責任
- `dinecore_orders`
  - 主訂單資料：`order_no`、`table_code`、`order_status`、`payment_status`、金額欄位。
- `dinecore_order_batches`
  - 批次資料：每次送單一個 submitted batch，下一批為 draft。
- `dinecore_cart_items`
  - 訂單品項明細：title、quantity、price、客製內容。
- `dinecore_order_timeline`
  - 狀態時間軸：顧客送單、櫃台改狀態、櫃台標記付款。
- `dinecore_table_sessions`
  - 掃碼 session 狀態（不是訂單主檔）：保存/失效顧客 token 與桌位關聯。

## 4. 1.0 核心規則
- 規則 A：訂單資料以 `dinecore_orders` 為唯一主體。
- 規則 B：櫃台列表與明細一律從 `orders + batches + cart_items` 組裝。
- 規則 C：付款完成只改 `payment_status=paid`，不產生帳冊寫入。
- 規則 D：付款完成後，對應 `guest_state_json` 僅需「失效」即可；是否物理刪除留待 2.0 決策。

## 5. 可執行步驟（按順序）
1. 確認送單後 DB 狀態
- 驗證 `dinecore_orders.payment_status = unpaid`。
- 驗證至少存在一筆 `dinecore_order_batches.status != draft`。
- 驗證 `dinecore_cart_items` 有該訂單品項。

2. 確認櫃台列表資料來源
- 後端 API `counterOrders` 僅以 `dinecore_orders` 查主資料。
- 批次/明細以關聯表補齊，不回頭以 table session 當主資料。

3. 實作付款標記（既有行為維持）
- 呼叫 `counterUpdatePaymentStatus`。
- 更新 `dinecore_orders.payment_status`。
- 對該訂單關聯 session 做 `expired`（或後續改為刪除策略）。

4. 前端狀態同步
- 付款後櫃台頁面即時顯示 `paid`。
- 顧客端若使用舊 token 查該訂單，應回應受限/失效訊息。

5. 驗收測試
- Case 1：送單成功後，櫃台可見未付款訂單。
- Case 2：標記付款後，櫃台顯示已付款。
- Case 3：付款後原 session 不可再繼續同單操作。

## 6. 2.0 待辦（本階段不做）
- 新增帳冊主表/明細表。
- 付款方式與交易編號落帳。
- 退款、折讓、沖銷流程。
- 日結與對帳報表。

## 7. 當前版本結論
- 目前系統可視為「訂單流 + 付款標記流」已成立。
- 1.0 先凍結在此規格，避免繼續混入帳務需求造成範圍膨脹。
