# 購物車及訂單流程前端頁面切換規則

## 0. 文件用途

本文件用於定義 DineCore 在「主單 + 多批次送單」模型下，前端各頁面在不同狀態切換時應如何反應。

目的：

- 明確規定前端何時刷新、何時清空、何時跳頁
- 避免購物車、結帳頁、訂單追蹤頁對同一批次有不同理解
- 讓輪詢與多瀏覽器共桌流程保持一致

---

## 1. 核心原則

前端必須遵守以下原則：

- 同一時間只允許一個 `currentDraftBatch`
- 前端不可自行假設某批次仍可編輯，必須以 API 回應為準
- 一旦發現 `draft batch` 已變更，必須立即切換上下文
- 已送單批次永遠只能唯讀，不可再回寫

---

## 2. 頁面角色分工

### Entry Page

用途：

- 取得桌號上下文
- 取得目前匿名顧客身份
- 取得目前 `currentDraftBatch`

責任：

- 建立進桌號的前端初始狀態
- 不承擔購物車或結帳編輯責任

---

### Menu Page

用途：

- 瀏覽商品
- 將商品加入目前 `draft batch`

責任：

- 只依附當前 `currentDraftBatch`
- 若 draft batch 已切換，必須無條件跟進新批次

---

### Cart Page

用途：

- 顯示目前 `draft batch`
- 輪詢共桌狀態
- 顯示目前參與顧客數
- 顯示已送出批次摘要

責任：

- 成為共桌協作狀態頁
- 當 A 送單後，B/C 應透過輪詢自動切到新 draft batch

---

### Checkout Page

用途：

- 顯示目前 `draft batch` 將送出的總結
- 送出目前批次

責任：

- 不可顯示舊批次為可編輯狀態
- submit 後不可停留在舊 draft 語意

---

### Order Tracker Page

用途：

- 顯示主單與所有已送出批次

責任：

- 以唯讀方式呈現歷史與目前狀態
- 不承擔 cart 編輯責任

---

## 3. 前端狀態模型草案

前端至少應能維持以下狀態：

- `orderId`
- `orderNo`
- `currentBatchId`
- `currentBatchNo`
- `currentBatchStatus`
- `participantCount`
- `orderingSessionToken`
- `orderingCartId`
- `submittedBatches`

說明：

- `currentBatch*` 用於操作上下文
- `submittedBatches` 用於唯讀顯示歷史

---

## 4. 頁面切換規則

### Rule 1：進桌號入口

當使用者進入 `/t/:tableCode`：

- 呼叫 `entry-context`
- 寫入：
  - `orderId`
  - `orderNo`
  - `currentBatchId`
  - `currentBatchNo`
  - `currentBatchStatus`
  - `orderingSessionToken`
  - `orderingCartId`

完成後才允許進入 menu / cart / checkout

---

### Rule 2：進 Menu Page

進入 `/t/:tableCode/menu` 時：

- 若缺少 `orderingSessionToken`
  - 回入口重新拿上下文

- 若 `currentBatchStatus !== draft`
  - 重新請求最新 `entry-context`

Menu Page 只允許對目前 `draft batch` 加商品

---

### Rule 3：進 Cart Page

進入 `/t/:tableCode/cart` 時：

- 先載入 cart payload
- cart payload 必須回：
  - `currentBatchId`
  - `currentBatchNo`
  - `currentBatchStatus`
  - `participantCount`
  - `submittedBatches`

Cart Page 顯示兩塊：

- `目前加點購物車`
- `已送出批次摘要`

---

### Rule 4：Cart Page 輪詢

Cart Page 每次輪詢時：

1. 呼叫 `GET /carts`
2. 讀取 API 回來的 `currentBatchId` / `currentBatchStatus`

若：

- `currentBatchId` 與前端目前記錄不同
- 或 `currentBatchStatus !== draft`

則必須：

- 清掉目前 editor / option draft / 本頁暫存編輯狀態
- 將頁面上下文切換到新的 draft batch
- 重新渲染目前購物車內容

不得：

- 繼續把舊批次當成可編輯內容

---

### Rule 5：A 送單成功後的本頁反應

當 A 在 checkout submit 成功後：

- 將本地 `currentBatchId` 視為已失效
- 依 response 取得：
  - `submittedBatchId`
  - `nextBatchId`
- 清空舊購物車編輯狀態
- 視產品體驗決定：
  - 跳成功頁
  - 或返回 cart 頁並顯示新 draft batch

---

### Rule 6：B 留在 Cart Page 時的反應

當 B 停留在 cart 頁，而 A 在另一瀏覽器送單成功：

下一次輪詢時，B 應該：

- 看到 `currentBatchId` 已更新
- 舊 batch 不再顯示為可編輯購物車
- 畫面提示：
  - `上一批已送出`
  - `目前可開始新一輪加點`

不得：

- 繼續保留舊 batch 內容在 editor 中
- 將新商品加進舊 batch

---

### Rule 7：Checkout Page 狀態失效處理

若使用者停留在 checkout page，而輪詢或 submit 前發現：

- `currentBatchStatus !== draft`
- 或 `currentBatchId` 已變更

則應：

- 中止送單按鈕
- 提示此批次已送出
- 重新整理上下文
- 視情況跳回 cart page 或成功頁

---

### Rule 8：Order Tracker 顯示規則

tracker 應分為：

- 主單摘要
- 批次列表

每個 batch 顯示：

- `batchNo`
- `status`
- `submittedAt`
- `items`
- `subtotal`

tracker 不應顯示可編輯控制元件

---

## 5. 元件層最小規則

### Menu Store

- 只操作 `currentDraftBatch`
- 若 API 回 `BATCH_LOCKED`
  - 立即重新抓上下文

### Cart Store

- 必須持有：
  - `currentBatchId`
  - `currentBatchNo`
  - `currentBatchStatus`
  - `participantCount`
  - `submittedBatches`

- 若 batch 改變：
  - 清 editor
  - 清 option sheet

### Checkout Store

- 只持有目前 `draft batch` summary
- submit 成功後應清掉舊批次上下文

---

## 6. UI 提示建議

建議補以下提示：

- `上一批已送出，已切換到新的加點批次`
- `目前共有 X 位顧客正在共桌點餐`
- `這一批已送出，請改在新的加點批次操作`

---

## 7. 驗證案例

### Case 1：同頁重整

- 同一瀏覽器重整 cart 頁
- 仍應維持同一位顧客與同一個 `draft batch`

### Case 2：A 送單，B 停留購物車

- B 下一輪 polling 後
- 應切到新的 draft batch

### Case 3：A 送單，B 停留結帳頁

- B 應被阻止再送出舊批次

### Case 4：新一輪加點

- B 在新 draft batch 加品項
- 應與上一批 submitted batch 清楚分離

---

## 8. 建議實作順序

1. Entry store 補 `currentBatch*`
2. Cart store 補 `currentBatch*`、`participantCount`、`submittedBatches`
3. Cart page 輪詢切換規則
4. Checkout page 失效處理
5. Tracker 顯示批次列表
