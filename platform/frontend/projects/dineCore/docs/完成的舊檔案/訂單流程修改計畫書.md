# dineCore 訂單流程修改計畫書

## 0. 文件目的
- 將 `dineCore` 收斂為較簡單、較乾淨的點餐架構。
- 主軸改為「瀏覽器暫存購物車，送單才落 DB」。
- 後端不再承擔草稿購物車狀態，只處理正式訂單、批次、追蹤與併單。
- 本文件取代先前以 `guest_state_json + DB 草稿購物車` 為中心的收斂方向。

## 1. 新方案總結
- 顧客未送單前的購物車資料只存在瀏覽器。
- 顧客送單時，前端將完整購物車 payload 一次送給後端。
- 後端在送單當下驗證商品、價格、選項、桌號與訂單狀態，並建立正式訂單資料。
- `cart/add-item`、`cart/change-item-quantity`、`cart/update-item`、`carts`、`checkout-summary` 這類草稿購物車 API 不再作為主流程必要 API。
- 併單改為正式訂單後處理 API，不與草稿購物車耦合。

## 2. 為何改走第一種方案

### 2.1 目標
- 結構簡單。
- 責任乾淨。
- 前端不需輪詢草稿購物車 API。
- 後端不必維護多人共享草稿狀態。

### 2.2 主要收益
- 移除 `guest_state_json` 這類共享草稿狀態容器。
- 顧客草稿 cart 不再進 DB，自然減少幽靈 cart item。
- 購物車頁與 checkout 頁不需要定期打 API 拉草稿。
- 後端職責集中在：
  - 建立正式訂單
  - 維護 submitted 後的 order / batch
  - tracker / staff / merge

### 2.3 代價
- 前端需接手草稿 cart 狀態、價格計算與暫存恢復。
- `checkout-submit` 需改成接收完整 cart payload。
- 送單時所有價格、商品有效性都要重新驗證。
- 若使用者清除瀏覽器資料或換裝置，未送單購物車不保留。

## 3. 核心設計原則
- 草稿購物車屬於前端狀態，不是後端狀態。
- 後端只保存正式送單後資料。
- 單機草稿優先，不做多人共享草稿購物車。
- 同桌多人若各自送單，之後用顯性併單處理。
- 顧客未送單前，不需要正式 `order_no`。

## 4. 新主流程

### 4.1 顧客點餐流程
1. 掃桌號進入。
2. 取得桌號可用性、菜單、必要的 ordering token。
3. 前端在瀏覽器建立本機 cart。
4. 顧客加減品項、修改規格、查看結帳摘要，都由前端本地處理。
5. 顧客按送單時，前端將完整 cart payload 送到 `checkout-submit`。
6. 後端驗證通過後，建立正式 `order / batch / cart_items / timeline`。
7. 顧客改看正式訂單的 success / tracker。

### 4.2 送單後再加點流程
1. 顧客已有一張已送單 order。
2. 顧客再次進入點餐頁後，本機可重新建立新的草稿 cart。
3. 顧客再次送單時，後端依規則：
   - 沿用原本的正式 order
   - 建立新的 submitted batch
4. staff / tracker 看到的是同一張 order 下多個 submitted batch。

### 4.2.1 再加點接回既有 order 的第一階段保守判定
- 第一階段不做複雜自動判斷，不依賴同桌多人共享草稿資訊。
- 顧客再次送單時，只有在以下條件同時成立時，才允許接回既有 order：
  - 同一 `table_code`
  - 同一有效 `orderingSessionToken`
  - 該正式 order 尚未 `paid`
  - 該正式 order 尚未 `cancelled`
  - 該正式 order 尚未 `merged`
- 若任一條件不成立，則建立新 order。
- 第一階段寧可多開新 order，再由後續併單處理，也不要錯接到他人的既有 order。

### 4.3 同桌多人流程
- 每個裝置各自維護自己的草稿 cart。
- 同桌顧客彼此不共享草稿內容。
- 同桌多人若各自送單，會形成多張正式 order。
- 若需一起結帳，再透過顯性併單 API 處理。

## 5. 前端責任

### 5.1 本機購物車
- `cart store` 改為本地狀態來源。
- 建議使用 `localStorage` 或 `sessionStorage` 保存：
  - `tableCode`
  - `orderingSessionToken`
  - cart items
  - customization
  - 本地小計資訊

### 5.1.1 本機 cart key / restore / 清空規則
- 本機 cart 必須以桌號與本機身份分開儲存，避免不同桌號互相污染。
- 建議 key 結構：
  - `dinecore.cart.{tableCode}.{orderingSessionToken}`
- restore 規則：
  - 進入桌號頁面後，若本機存在同桌同 token 的 cart，則還原
  - 若桌號不同，絕不共用舊 cart
  - 若 token 改變，舊 cart 不自動接續
- 清空規則：
  - 送單成功後，清空該 key 的本機 cart
  - 使用者主動清空購物車時，清空該 key
  - 若本機 cart 超過保存期限，可在 entry 或 cart 初始化時清除
- 建議本機 cart 內容至少包含：
  - items
  - note / customization
  - orderingLabel
  - lastSavedAt
  - 可選的前端 summary cache

### 5.2 本地 checkout summary
- checkout 頁直接使用本地 cart 計算：
  - item count
  - subtotal
  - service fee
  - tax
  - total

### 5.3 前端仍需呼叫的 API
- `entry-context`
- `menu`
- `checkout-submit`
- `checkout-success`
- `order-tracker`
- staff 相關 API
- 未來的 merge API

### 5.4 可淘汰或降級的 API
- `GET /api/dinecore/carts`
- `POST /api/dinecore/cart/add-item`
- `POST /api/dinecore/cart/change-item-quantity`
- `POST /api/dinecore/cart/update-item`
- `GET /api/dinecore/checkout-summary`

### 5.5 輪詢調整
- cart 頁不再需要為草稿購物車每 5 秒或 10 秒打 API。
- checkout 頁在送單前也不需要輪詢草稿資料。
- 只有正式訂單 tracker 才保留輪詢或重整需求。

## 6. 後端責任

### 6.1 後端職責邊界
- 後端不再負責保存未送單購物車。
- 後端只處理：
  - 桌號/點餐資格驗證
  - 菜單資料提供
  - 正式送單建立
  - 正式訂單追蹤
  - 併單
  - staff 作業

### 6.2 `checkout-submit` 新責任
- 接收完整 cart payload。
- 驗證：
  - table 是否可點餐
  - menu item 是否存在
  - menu item 是否售完或隱藏
  - customization 是否合法
  - price 是否與後端重新計算一致
- 驗證成功後：
  - 建立或續用 order
  - 建立 submitted batch
  - 寫入 cart items
  - 更新 order totals
  - 寫入 timeline

### 6.2.1 `checkout-submit` 原子交易規則
- `checkout-submit` 必須在單一資料庫交易內完成。
- 交易內至少包含以下步驟：
  1. 驗證桌號與點餐權限
  2. 驗證 payload items 與 customization
  3. 決定建立新 order 或接回既有 order
  4. 建立本次 submitted batch
  5. 寫入本次 batch 對應 cart items
  6. 重算並更新 order totals / order status
  7. 寫入 timeline
- 任一步驟失敗，整筆交易回滾，不得留下半成品 order / batch / items。
- 前端只有在後端交易成功回應後，才能清空本機 cart。
- 後端不得信任前端傳入的最終總價為唯一依據，必須自行重算。

### 6.2.2 `checkout-submit` 防重複提交規則
- 前端送單按鈕在第一次提交後必須立即進入 `submitting` 狀態並禁止重按。
- 在同一次提交流程完成前，前端不得再次送出相同 cart。
- 後端需具備最小限度的冪等防護，避免連點、重送、重試造成重複建立訂單或 batch。
- 第一階段建議前端送單時帶入 `clientSubmissionId`。
- 後端應以 `orderingSessionToken + clientSubmissionId` 作為短時間窗口內的重複提交判斷鍵。
- 若第一次提交已成功，後續相同提交應回傳同一筆成功結果，或回傳可識別的重複提交結果。
- 送單失敗時，本機 cart 必須保留，不得提前清空。
- 只有在後端明確回傳成功後，前端才清空本機 cart 並跳轉 success。

### 6.2.3 `checkout-submit` 成功 / 重複成功統一回應規格
- 無論是首次成功，或因重複提交而命中同一次成功結果，回應格式都應一致。
- 第一階段建議統一回傳：
  - `orderId`
  - `orderNo`
  - `batchId`
  - `batchNo`
  - `orderStatus`
  - `paymentStatus`
  - `submittedAt`
  - `isDuplicateSubmission`
- 規則：
  - 首次成功：`isDuplicateSubmission = false`
  - 重複成功：`isDuplicateSubmission = true`
- 前端不應因 `isDuplicateSubmission` 不同而走不同流程，兩者都視為可安全清空本機 cart 並導向 success。

## 7. `order` 與 `batch` 規則

### 7.1 `order` 的責任
- `order` 表示顧客在該桌號下的一張正式訂單容器。
- `order` 只在第一次成功送單時建立。
- 未送單前不建立正式 `order_no`。

### 7.2 `batch` 的責任
- `batch` 表示一次送單提交。
- 每次送單產生一個新的 submitted batch。
- batch 是 order 的附屬資料，不是前端草稿購物車。

### 7.3 `order.status` 與多個 `batch` 的關係
- `order_status` 應反映整張 order 的整體履約狀態。
- `batch.status` 應反映單一送單批次狀態。
- 一張 order 可有多個 submitted batch。
- staff / tracker 應以 order 為主體、batch 為明細。
- `order_status` 的推導邏輯必須集中定義。

### 7.3.1 `order_status` 集中推導規則
- `order_status` 不可由各 controller 各自自由推論。
- 必須集中在單一規則層或 helper 中推導，staff / tracker / reports 共用。
- 第一階段建議推導優先序：
  1. 若 order 已被併入他單，則為 `merged`
  2. 若 order 已取消，則為 `cancelled`
  3. 若 `payment_status = paid` 且訂單履約完成，則為 `completed`
  4. 若存在任一有效 batch 為 `ready`，則 order 可為 `ready`
  5. 若存在任一有效 batch 為 `preparing`，則 order 可為 `preparing`
  6. 若至少存在一筆 submitted batch，則 order 為 `pending`
- `draft` 不再作為長期正式狀態，只保留給送單交易中的短暫中間態，或後續直接移除。

### 7.3.2 有效 batch 定義
- 有效 batch 指已正式屬於某張 order 履約流程、應參與 staff / tracker / status 推導的 batch。
- 第一階段下，有效 batch 至少需滿足：
  - 已成功寫入資料庫
  - 屬於某張正式 order
  - 狀態不為送單交易失敗中間態
- 第一階段建議將以下視為有效 batch：
  - `submitted`
  - `preparing`
  - `ready`
  - `completed`
- 以下不應視為有效 batch：
  - 送單交易失敗殘留的中間資料
  - 未成功提交完成的暫存 batch
- `cancelled` batch 是否計入有效 batch，需依 staff 與報表需求決定；第一階段建議：
  - 可作歷史資料保留
  - 但不計入有效履約中的 batch

## 7.4 前端價格與後端重算的責任分界
- 前端價格計算的責任：
  - 即時顯示使用者目前看到的預估金額
  - 支援 cart / checkout 畫面體驗
  - 讓使用者在送單前知道大致金額
- 後端價格計算的責任：
  - 作為唯一正式金額來源
  - 依據當下有效菜單、選項、加價規則重新計算
  - 決定正式寫入 `orders / cart_items` 的金額
- 責任邊界：
  - 前端價格是預估與展示
  - 後端價格是正式與結算
- 若前端計算與後端重算不一致：
  - 以後端為準
  - 可回傳錯誤碼或正規化後的價格結果，交由前端提示使用者

## 8. session 與過期規則

### 8.1 session 的角色
- session 只代表顧客本次點餐身份。
- session 不再承載草稿購物車內容。
- session 過期只影響草稿身份，不影響已送單正式資料。

### 8.2 session 過期後的行為
- session 過期後，顧客無法直接以原本本機草稿繼續送單。
- 若該顧客已送出正式 order，tracker 仍可查看正式訂單。
- 若要再次加點，應重新建立有效 session，再由後端決定是否接到既有 order。

### 8.3 第一階段保守規則
- session 過期後不自動恢復原 order 的加點權限。
- 顧客若重新進入並要加點，可由後端依桌號與權限決定：
  - 接回既有 order
  - 或建立新 order，再由 staff 視需要併單

## 9. cart_id 的語意
- 若第一種方案完全落地，`cart_id` 不再需要承擔共享購物車語意。
- `cart_id` 若保留，只是送單 payload 與 item 群組的技術欄位。
- 長期可評估直接移除 `cart_id`，改以 payload item list 與 session / order 對應。

## 10. 有效訂單判定規則

### 10.1 staff 列表可見規則
- staff 正式列表只顯示有效正式訂單。
- 有效正式訂單的最低條件是：
  - 至少存在一筆 submitted batch
- 未送單草稿不屬於 staff 訂單資料。

### 10.2 tracker 顯示規則
- tracker 只顯示正式送單後的訂單。
- 純前端草稿不進 tracker。

### 10.3 併單候選規則
- 候選只列同桌、有效、未付款、未取消、未 merged 的正式訂單。
- 草稿購物車不在併單範圍內。

## 11. cleanup 策略

### 11.1 新方案下 cleanup 的範圍
- 因草稿購物車不進 DB，cleanup 壓力會明顯降低。
- cleanup 主要改為處理：
  - 過期 session
  - 已完成後需封存或搬移的正式訂單
  - 少量送單中斷造成的不完整資料

### 11.2 產品說明
- cleanup 是延遲清理，不是即時清理。
- cleanup 由請求低頻觸發，不依賴 scheduler。
- 第一階段只標記與忽略，不立即實體刪除。

### 11.3 是否需要新增 cleanup control table
- 若仍採 request-triggered cleanup，建議保留一張 housekeeping / cleanup control table。
- 這是第二方案與第一方案都可共用的基礎設施。
- 若短期資料量極小，也可先不做，改為 staff 手動清理。

## 12. 併單策略

### 12.1 定位
- 併單是正式訂單後處理功能。
- 不與草稿購物車耦合。

### 12.2 第一階段做法
- 各自送單。
- staff 或顧客端顯性按鈕列出同桌有效正式訂單。
- 選擇主單與被併單後執行 merge API。

### 12.3 被併單顯示規則
- 被併單應標記為 `merged` 或等價狀態。
- staff 主列表不再把被併單當可操作正式訂單。
- tracker 不再把被併單顯示為進行中獨立訂單。
- 若需保留可見性，只顯示「已併入主單」提示。

## 13. 後端資料庫比對與修改建議

### 13.1 比對結論
- 目前資料庫是以「後端保存草稿購物車」為前提設計。
- 第一種方案改成「瀏覽器暫存購物車，送單才落 DB」後，資料庫不需要再承擔草稿 cart/session 的持久化責任。
- 因此本次重構重點不在大量新增資料表，而在：
  - 刪除不再有語意的草稿資料結構
  - 弱化目前只為草稿而存在的欄位與查詢
  - 保留正式訂單、批次、歷程、staff、報表相關資料表

### 13.2 資料表逐項建議

#### A. 保留，且仍是核心資料表

`dinecore_tables`
- 保留。
- 功能不變。
- 仍作為桌號、區域、可點餐狀態的主表。

`dinecore_menu_categories`
- 保留。
- 功能不變。

`dinecore_menu_items`
- 保留。
- 功能不變。
- `default_option_ids_json`、`option_groups_json` 可繼續存在，因為它們是菜單定義，不是共享狀態。

`dinecore_orders`
- 保留。
- 但語意需改為「正式訂單主表」。
- 不再鼓勵建立未送單 draft order。
- 建議修改：
  - `order_status` 預設值不再依賴草稿流程。
  - `order_no` 應在首次送單成功後才產生。
- 建議文件層與實作層同步調整：
  - 若仍保留 `draft` 狀態，僅用於送單交易進行中或極短暫中間態，不再當作長期草稿狀態。

`dinecore_order_batches`
- 保留。
- 仍是核心資料表。
- 但語意改為「每次正式送單產生一筆 batch」。
- 建議修改：
  - `status` 預設 `draft` 的必要性要重新評估。
  - 若前端草稿完全不落 DB，則 batch 建立時點可延後到送單當下。
- 可接受作法：
  - 送單交易開始時先建立 batch，成功後直接為 `submitted`。
  - 或建立即是 `submitted`，不保留資料庫層 draft batch。

`dinecore_cart_items`
- 保留。
- 但語意改為「已送單明細」，不再是草稿購物車表。
- 建議修改：
  - 不再由 `cart/add-item`、`change-item-quantity`、`update-item` 即時寫入。
  - 改為 `checkout-submit` 成功時一次寫入。
- 欄位評估：
  - `order_id`：保留
  - `batch_id`：保留
  - `table_code`：可保留，方便查詢
  - `menu_item_id`：保留
  - `title / quantity / price / note / options_json / selected_option_ids_json`：保留
  - `cart_id`：長期可評估移除，短期可先保留以降低 migration 風險

`dinecore_order_timeline`
- 保留。
- 功能不變。
- 仍作為正式訂單狀態歷程。

`dinecore_cart_items` 正式明細語意
- 本表在第一種方案下必須明確定義為「正式送單明細表」。
- 每一筆 `dinecore_cart_items` 都必須對應已成功提交的 batch。
- 未送單草稿不得寫入本表。
- 顧客在 cart 頁面加減品項、改規格時，不得直接寫本表。
- 本表唯一主要寫入入口應為：
  - `checkout-submit` 成功後的正式落庫
  - 或受控的正式訂單後處理流程
- staff、tracker、報表都應將本表視為正式資料來源，而非暫存資料來源。

`dinecore_staff_profiles`
- 保留。
- 功能不變。

`dinecore_business_closings`
- 保留。
- 功能不變。

`dinecore_business_closing_history`
- 保留。
- 功能不變。

#### B. 弱化或改寫語意的資料表

`dinecore_table_sessions`
- 現況：
  - 一桌一筆 row
  - `guest_state_json` 承載多人 session 狀態
- 第一種方案下：
  - 不再需要用它保存草稿購物車 session
  - 不再需要 `guest_state_json`
- 建議方向二選一：

方案 B1：弱化保留
- 保留此表，只作為「桌號最近互動 / 正式訂單上下文」輕量資訊表。
- 移除 `guest_state_json`。
- `order_id` 改為可選輔助欄位，不再當草稿綁定主鍵。
- 第一階段建議先走 B1，避免一次拔掉所有 session 相依。
- 第一階段下 `dinecore_table_sessions` 只保留：
  - 桌號最近一次合法進場資訊
  - `orderingSessionToken / orderingLabel` 的輕量上下文
  - 是否可接回既有正式 order 的輔助判斷
- 第一階段下 `dinecore_table_sessions` 不再承擔：
  - 多人共享草稿狀態
  - 草稿 cart 持久化
  - `guest_state_json` session 容器

方案 B2：完全退出主流程
- `table_sessions` 不再參與顧客點餐主流程。
- 桌號可點餐判斷只看 `dinecore_tables`。
- 顧客身份直接靠前端 token + 正式送單驗證。
- 若採此方案，後續可考慮整表停用。

建議：
- 若目標是最乾淨，優先採 B2。
- 若想降低改動風險，可先採 B1，等流程穩定再移除。

#### C. 可刪除或停止使用的資料表

`dinecore_guest_sessions`
- 目前 schema 有這張表，但現有 controller 幾乎未以它作主流程資料來源。
- 在第一種方案下，顧客草稿 session 存於瀏覽器，本表更沒有存在必要。
- 建議：
  - 不再使用
  - 若線上無依賴，可列入刪除候選

### 13.3 建議新增的資料表

`housekeeping / cleanup control table`
- 建議新增 1 張表。
- 用途：
  - 記錄 request-triggered cleanup 的執行頻率與鎖定狀態
- 建議欄位：
  - `job_key`
  - `last_run_at`
  - `locked_until`

`merge record table`
- 正式需求。
- 需新增 1 張併單紀錄表。
- 用途：
  - 記錄主單
  - 記錄被併單
  - 記錄操作者
  - 記錄操作時間
  - 記錄原因或備註
- `timeline` 可作顯示用途，但不得取代正式 merge record table。

### 13.4 欄位層級建議

  - 由「建草稿就先產生」改為「首次送單成功才產生」

  - 保留，由送單成功後計算寫入

#### `dinecore_cart_items`

  - 第一種方案下語意弱化
  - 可先保留，第二階段再決定是否移除

### 13.5 舊 cart API 與舊 session 依賴盤點

#### 舊 cart API
- `GET /api/dinecore/carts`
  - 舊責任：從 DB 組裝草稿 cart payload
  - 新方案：由前端本機 cart 取代，主流程停用
- `POST /api/dinecore/cart/add-item`
  - 舊責任：新增草稿 cart item 到 DB
  - 新方案：由前端本機 store 處理，API 停用
- `POST /api/dinecore/cart/change-item-quantity`
  - 舊責任：直接更新 DB 草稿數量
  - 新方案：由前端本機 store 處理，API 停用
- `POST /api/dinecore/cart/update-item`
  - 舊責任：直接更新 DB 草稿 customization
  - 新方案：由前端本機 store 處理，API 停用
- `GET /api/dinecore/checkout-summary`
  - 舊責任：從 DB 草稿計算 summary
  - 新方案：由前端本地計算，API 停用

#### 舊 session 依賴
- `resolveOrderingSession()`
  - 舊責任：建立/恢復草稿 order、草稿 session、共享狀態
  - 新方案：保留最小身份驗證與送單合法性判斷
- `guest_state_json`
  - 舊責任：承載同桌多人 session 狀態
  - 新方案：移除
- `findTableSessionForUpdate()` / `persistGuestStateForTableSession()` / `listSessionsForOrder()`
  - 舊責任：從 `table_sessions + guest_state_json` 反查 order 與顧客
  - 新方案：退出草稿主流程依賴
- staff 端 `closeTableSessionsForOrder()` / `ensureActiveTableSessionForOrder()`
  - 舊責任：配合草稿 session 同步 table 狀態
  - 新方案：改為正式訂單上下文輔助邏輯或弱化保留

### 13.6 migration 建議順序
1. 先停用草稿 cart API 的寫入路徑。
2. 重寫 `checkout-submit`，改為送單成功才寫 `orders / batches / cart_items`。
3. `order_no` 改為送單後產生。
4. `dinecore_table_sessions` 從主流程中退出或移除 `guest_state_json`。
5. 視實際上線穩定度，再刪除不再使用的 table / column。

### 13.7 不建議立即刪除的項目
- `cart_id`
  - 雖然語意弱化，但短期保留可以降低資料搬移風險。
- `dinecore_table_sessions`
  - 除非確認所有 staff / tracker / auth flow 都不再依賴，否則先弱化、後移除。
- `draft` 相關 enum / status
  - 先停止主流程使用，再清 schema。

## 15. 分階段執行建議

### Phase 1：前端本地購物車
- cart store 改為本地狀態。
- checkout summary 改為前端計算。
- cart / checkout 頁移除草稿 API 依賴。

### Phase 2：重寫 `checkout-submit`
- 改收完整 cart payload。
- 建立送單驗證與正式落庫邏輯。

### Phase 3：精簡後端草稿 API
- 停用或移除 cart 相關 API。
- 精簡 controller 內 DB 草稿依賴。

### Phase 4：正式訂單後處理
- 補強 tracker。
- 補清理機制。
- 新增按鈕式併單。

## 16. 風險
- 送單時驗證邏輯若不完整，前端價格可能與後端重算不一致。
- 本地購物車若設計不當，刷新或切裝置會造成顧客體驗斷裂。
- 既有 cart API 仍被其他頁面依賴時，重構會造成回歸風險。

## 17. 完成定義（DoD）
- 草稿購物車不再落 DB。
- cart / checkout 草稿流程不依賴草稿 API 輪詢。
- 正式訂單只在送單成功後建立。
- `checkout-submit` 成為正式訂單建立唯一入口。
- tracker / staff / merge 只操作正式送單資料。
- 前後端責任邊界清楚：前端管草稿，後端管正式訂單。
