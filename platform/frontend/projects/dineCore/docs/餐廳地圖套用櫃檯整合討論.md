# 餐廳地圖套用櫃檯整合討論

## 文件定位
本文件用於討論 DineCore 中「將餐廳地圖套用至櫃檯，整合桌號、點餐與訂單管理」的新模組方向。

## 目前共識
- 不直接修改既有已完成模組的核心邏輯。
- 以新模組方式承接地圖與櫃檯整合功能。
- 地圖編輯資料與營運訂單資料必須維持清楚邊界。
- 需透過轉換層或映射層整合桌號、地圖桌位與訂單資料。

## 待討論問題
- 新模組的正式名稱與定位。
- 新模組與既有 `counter`、`table-admin`、`restaurant-map-editor` 的邊界。
- 桌號、桌位、訂單之間的對應規則。
- 第一階段最小可行整合範圍。
- 是否只先做櫃檯讀取地圖與查看桌況，不先做雙向編輯。

## 討論紀錄
- 2026-03-20：決定先以新模組方向討論，避免動到現有已完成模組。

## 模組目標
本模組的目標，不是取代現有櫃檯或桌位管理模組的全部責任，而是新增一個「以餐廳地圖為主視圖」的櫃檯整合入口，讓使用者可以在地圖上直接查看桌位分布、桌況狀態、點餐進度與訂單操作入口。

此模組應作為餐廳地圖產生器完成後的下一階段應用層，而不是反向修改地圖編輯器本身，也不是把既有櫃檯模組直接改造成地圖編輯器。

## 核心原則
- 不直接修改既有已完成模組的核心邏輯。
- 不讓既有 `counter`、`table-admin`、訂單流程模組直接依賴地圖編輯器內部 state。
- 地圖資料與營運資料分流處理。
- 以新模組承接整合責任，透過轉換層與既有模組對接。
- 舊模組先保留，待新模組驗證穩定後，再評估替換或淘汰。

## 為什麼要新開模組
若直接在既有櫃檯模組或桌位模組內擴寫地圖功能，容易產生以下問題：

- 原本已穩定的訂單流程被新功能污染。
- 桌位資料與地圖編輯資料語意混在一起。
- 一旦地圖資料格式調整，既有模組也被迫一起修改。
- 舊流程與新流程交錯後，問題難以定位。

因此本功能應獨立成新模組，將「地圖視圖整合」視為新的應用層，而不是插入既有模組核心。

## 新模組定位
此模組可暫定為：

- 櫃檯地圖整合模組
- 或：地圖式櫃檯工作台
- 或：桌況地圖櫃檯模組

其責任應為：

- 載入餐廳地圖正式資料。
- 載入桌位資料與桌位對應資訊。
- 載入訂單與桌況狀態。
- 將桌位狀態顯示於地圖上。
- 提供從桌位節點進入點餐、訂單、桌況操作的入口。

其不負責：

- 地圖編輯。
- 地圖物件建立或修改。
- 取代既有訂單系統完整流程。
- 直接改寫既有桌位管理模組內部資料結構。

## 與既有模組的邊界

### 1. `restaurant-map-editor`
負責地圖與桌位的編輯、儲存、重新載入。

本次新模組只使用其輸出的正式資料，不直接依賴其編輯中暫存狀態，不依賴 `draftState` 作為營運顯示基礎。

### 2. `table-admin`
目前若已承擔桌號、QR、桌位基本管理等責任，則維持原責任不變。

新模組不直接把 `table-admin` 改造成地圖模組，而是透過映射規則把桌號資料套到地圖桌位節點上。

### 3. `counter`
既有櫃檯模組維持原本訂單管理責任。

新模組可視為櫃檯的新入口視圖，但不應在第一階段直接重寫既有 `counter` 邏輯。較合理做法是從地圖節點導向既有櫃檯流程，或讀取既有櫃檯資料後做地圖顯示。

## 資料邊界

### 地圖層資料
由餐廳地圖產生器提供：

- `map`
- `objects`
- `tables`

其中：
- `objects` 用於場域圖形顯示。
- `tables` 用於桌位定位與地圖上的桌位節點。

### 營運層資料
由櫃檯與訂單系統提供：

- 桌號
- 桌況
- 訂單狀態
- 付款狀態
- 使用中 / 空桌 / 已結帳等營運資訊

### 對接層資料
需新增 mapping / adapter layer，負責：

- 將地圖中的 `table` 對應到營運桌號。
- 將訂單資料掛到對應桌位節點。
- 處理未綁定桌位、重複桌號、缺漏資料等狀況。

## 第一階段最小可行範圍
第一階段不應一次做到完整櫃檯替換，應先收斂為：

- 可載入正式地圖。
- 可在地圖上看到桌位節點。
- 可將桌位節點對應到桌號。
- 可在桌位上顯示基本狀態。
- 點擊桌位後，可進入該桌位的訂單或櫃檯操作入口。

第一階段先不做：

- 地圖內直接編輯桌位。
- 地圖內直接修改場域圖形。
- 取代全部既有櫃檯頁面。
- 複雜的雙向同步編輯。

## 待下一步確認
- 新模組正式命名。
- 桌位對應是否以 `label`、`tableCode`、或新增專用對接欄位為主。
- 第一階段櫃檯入口是「新頁面」還是「現有 counter 的延伸入口」。
- 地圖節點點擊後，是開啟側邊詳情、跳轉頁面，還是彈出操作面板。

## 畫面顯示原則
考量櫃檯使用效率與前端效能，第一階段不使用地圖縮圖清單。

地圖切換區僅顯示：
- 地圖名稱
- 必要時可加上區域或樓層文字標示

使用者點擊名稱後，主畫面切換顯示對應地圖。

此做法可先降低渲染成本，並保留後續再擴充縮圖預覽的可能。

## 桌位訂單上限定義
桌位需支援一個可由地圖產生器設定的欄位，用於限制該桌位在未結帳前可同時存在的訂單數。

建議第一階段欄位語意如下：

- `maxActiveOrders`

定義：
- 代表此桌位在未結帳前，同一時間允許存在的有效訂單數上限。
- 若目前未結帳訂單數已達上限，則不應再允許新增新的訂單。
- 此欄位屬於桌位結構設定，應在餐廳地圖產生器中定義與維護。
- 櫃檯整合模組僅負責讀取此限制並在業務流程中套用，不負責編輯此值。

補充原則：
- 第一版可先將此欄位視為桌位容量與有效訂單上限的主要依據。
- 若未來需要區分「座位數」與「可同時存在訂單數」，再拆分為不同欄位。
- 第一階段先不在櫃檯整合模組內修改桌位容量類設定。

## 欄位語意收斂補充
第一階段不定義「幾人桌」、「座位數」或其他容量型桌位模型。

原因如下：
- 本專案目前要處理的是訂單承載上限，而不是座位容量管理。
- 若提前引入幾人桌語意，容易與後續營運規則混淆。
- 不做過多桌位定義，可降低資料模型複雜度與邏輯混亂風險。

因此第一階段只保留：
- `maxActiveOrders`

其唯一語意為：
- 此桌位在未結帳前，同一時間最多允許存在幾張有效訂單。

補充規則：
- 第一階段不得將 `maxActiveOrders` 解釋為座位數。
- 第一階段桌位模型不延伸為幾人桌、桌型、容納人數等結構。
- 若未來真的需要處理座位容量，再新增獨立欄位與獨立規則，不回頭混用 `maxActiveOrders`。

## 使用者語言與系統欄位分離
第一階段可對使用者以「幾人桌」語意呈現，讓櫃檯或設定人員更容易理解此桌位限制。

例如：
- `2 人桌`
- `4 人桌`
- `這桌最多可開 2 張單`

但系統內部資料欄位仍維持：
- `maxActiveOrders`

原因如下：
- 使用者語言應以直觀易懂為優先。
- 系統欄位語意應以精準與穩定為優先。
- 避免將「座位容量」與「有效訂單限制」在資料模型層直接混為同一概念。

補充規則：
- UI 可用「幾人桌」作為顯示或輸入語意。
- 內部落地資料仍以 `maxActiveOrders` 為準。
- 第一階段若 UI 提供 `2 人桌`、`4 人桌` 之類選項，實際上應映射為對應的 `maxActiveOrders` 值。

## 現有訂單模型沿用原則
第一階段地圖套用櫃檯整合，沿用現有桌位與訂單模型，不重構既有訂單核心流程。

目前系統的主軸為：
- 桌位以 `table_code` 作為主要對接鍵。
- 一張桌位同一時間以單一 active order 為主。
- 同桌後續加點與流程延伸，透過既有 order / batch 機制處理。

本階段整合原則如下：
- 不新增多主單並行模型。
- 不在地圖整合模組內重寫 guest session、table session、checkout 流程。
- 不變更既有 `counter`、`guest ordering`、`table session` 的核心資料關係。
- 地圖整合模組只讀取既有桌位與訂單資料，作為定位、顯示與業務入口。

## 訂單拆分與互不影響原則
雖然沿用既有模型，但仍需確保資料責任拆分清楚，避免整合後互相污染。

需維持以下原則：
- 地圖資料只負責場域與桌位定位，不直接承接訂單流程狀態寫入。
- 訂單資料仍由既有訂單系統維護，不回寫成地圖編輯資料。
- 地圖整合模組僅作為桌位對應訂單的讀取與操作入口。
- 桌位顯示狀態應來自既有訂單與桌位 session 結果，而不是在地圖模組內另造一套狀態來源。

可接受的整合結果為：
- 原有訂單功能維持完整可用。
- 地圖模組只是在櫃檯提供更直觀的操作入口。
- 不同桌位的訂單資料彼此獨立，不因地圖整合而互相影響。

## 最穩健導入方案
若目標是在不破壞現有已完成模組的前提下，將餐廳地圖導入櫃檯與訂單管理，最穩健的方式不是重寫既有系統，而是讓新模組以既有資料結構為基礎進行套接。

### 1. 以 `table_code` 作為唯一對接鍵
目前既有桌位、訂單、guest session、table session 的主關聯鍵皆為 `table_code`。

因此第一階段應採用：
- 地圖桌位新增對接欄位，例如 `tableCode`
- `tableCode` 對應既有 `dinecore_tables.code`
- 櫃檯地圖模組透過 `tableCode` 讀取既有桌位、session 與訂單資料

補充原則：
- 不以地圖桌位 `label` 作為正式對接鍵。
- 不以地圖編輯器內部 `table id` 對接既有訂單系統。
- `label` 可作為顯示用途，但不可作為唯一穩定關聯鍵。

### 2. 新模組只讀正式地圖
櫃檯整合模組只讀取正式儲存的地圖資料。

- 正式地圖作為營運顯示基礎。
- 草稿資料只存在於餐廳地圖編輯器模組。
- 櫃檯模組不依賴 `draftState`。

### 3. 第一階段作為地圖式入口，不重寫櫃檯流程
第一階段不應將所有櫃檯業務動作搬進新模組，而應先作為：
- 地圖定位入口
- 桌位狀態顯示入口
- 訂單列表與既有櫃檯操作流程入口

建議互動：
- 使用者點擊桌位圖示
- 顯示該桌位目前相關的訂單列表或 active order
- 再導向既有訂單詳情或業務操作流程

### 4. 以 adapter / view model 層整合資料
新模組不應直接將地圖格式與既有櫃檯資料混在一起。

應建立一層對接轉換：
- 輸入：正式地圖資料 `map + tables`
- 輸入：既有桌位、table session、order 資料
- 輸出：櫃檯地圖使用的 view model

此做法可讓：
- 地圖格式調整時，不直接衝擊櫃檯模組
- 櫃檯顯示需求變動時，不反向污染地圖資料模型
- 未來若替換櫃檯頁面，也能沿用同一層轉換規則

### 5. 未綁定桌位不得直接參與業務操作
若地圖桌位尚未對應到既有 `table_code`，則：
- 只能顯示為未綁定桌位
- 不應開啟訂單列表
- 不應直接進入櫃檯業務操作流程

此規則可避免錯誤對接造成桌位與訂單資料混亂。

### 6. 導入順序應分階段
建議導入順序如下：
- 載入正式地圖
- 建立桌位與 `table_code` 綁定
- 顯示桌位狀態
- 點擊桌位顯示訂單列表
- 從地圖入口接入既有櫃檯操作流程
- 最後才評估是否將更多櫃檯操作內嵌到地圖頁

## 導入策略總結
第一階段最穩健的方式是：
- 不讓舊系統配合地圖重構
- 讓新地圖模組以 `table_code` 為橋接鍵
- 讀取並包裝現有桌位與訂單系統
- 先完成「地圖式入口與桌位狀態顯示」
- 再逐步增加業務操作深度

## 桌位正式欄位預留原則
地圖產生器中的桌位正式資料，欄位可以先預留，但不能在真正需要時才發現資料結構不存在。

因此本階段原則為：
- 欄位可以先做，但不一定要第一階段全部啟用。
- 欄位可以先存在，但第一階段只使用責任已明確定義的欄位。
- 不得因為欄位已預留，就提前在多個模組中任意混用其語意。

## 第一階段建議桌位正式欄位
第一階段桌位正式資料建議至少包含：

- `id`
- `x`
- `y`
- `width`
- `height`
- `label`
- `tableCode`
- `maxActiveOrders`

欄位說明：
- `id`：地圖編輯器內部桌位識別。
- `x / y / width / height`：桌位在地圖中的世界座標與尺寸。
- `label`：地圖上的顯示名稱。
- `tableCode`：對接既有 `dinecore_tables.code` 的正式橋接欄位。
- `maxActiveOrders`：此桌位在未結帳前同時可存在的有效訂單數上限。

## 可預留但第一階段不作為正式對接依據的欄位
以下欄位可先預留，但第一階段不應作為核心業務判定依據：

- `areaName`
- `floorName`
- `status`
- `orderingEnabled`
- `meta`

補充原則：
- 上述欄位可作為未來擴充保留。
- 第一階段若需顯示營運狀態，仍應以既有桌位與訂單系統資料為主。
- 地圖桌位資料不應在第一階段直接取代既有營運主資料來源。

## 正式地圖到桌號管理的導入流程
正式地圖儲存後，不應直接自動將桌位資料寫入資料庫。

原因如下：
- 地圖變更不應立即衝擊既有桌號與營運資料。
- 若地圖編輯時刪除、改名、移動桌位，自動同步容易造成營運資料混亂。
- 桌號管理屬於營運主資料，導入主動權應保留在使用者手上。

因此本階段採用以下原則：
- `restaurant-map-editor` 只負責產出正式地圖。
- 正式地圖中的桌位資料作為可導入來源。
- 是否將桌位寫入 `dinecore_tables`，由使用者在 `table-admin` 中主動決定。

## 導入責任分工
### `restaurant-map-editor`
- 負責儲存正式地圖 JSON。
- 保留地圖桌位與對接欄位資料。
- 不直接修改桌號資料庫。

### `table-admin`
- 提供正式地圖清單或區域清單。
- 讓使用者選擇要查看的正式地圖。
- 顯示該地圖內的桌位清單。
- 由使用者決定哪些桌位要導入、綁定、忽略或建立新桌號。
- 實際寫入 `dinecore_tables` 與相關營運資料。

### 新櫃檯地圖模組
- 只讀取正式地圖與已導入的桌號主資料。
- 不負責桌位導入與桌號建立。

## table-admin 應新增的導入能力
桌位管理模組應新增一個「地圖桌位導入區」，至少包含：

- 正式地圖清單
- 區域或樓層名稱
- 該地圖的桌位列表
- 每筆桌位目前的導入狀態

每筆桌位至少應有以下可能狀態：
- 尚未建立對應桌號
- 已綁定既有 `table_code`
- 已建立新桌號資料
- 已忽略

## 導入策略結論
正式地圖是桌位來源，但不是桌號主資料。

因此：
- 地圖變更不應自動重設營運桌位資料。
- 桌位是否導入、如何導入、何時導入，應由使用者在桌位管理模組中主動控制。
- 導入主動權放在使用者手上，才能避免地圖一變就造成整批桌位資料失效。

## 顯示編號與內部識別必須分離
桌位若只依賴可變動的顯示編號，當桌位刪除、重排或重新編號時，會直接造成 QR、桌號綁定與訂單對接混亂。

因此本系統必須明確分離三種概念：

- `id`
- `label`
- `tableCode`

### `id`
- 作為地圖桌位的內部唯一識別。
- 一旦建立後不可因排序、刪除其他桌位、重新編號而改變。
- 為最保險的內部設定依據。

### `label`
- 作為畫面顯示用途。
- 可依需求重新排序、重新編號或調整顯示文字。
- 不可作為正式對接訂單、QR 或桌號主資料的唯一鍵。

### `tableCode`
- 作為對接既有 `dinecore_tables.code` 的正式營運桌號。
- 用於串接桌位管理、QR、訂單與 session。
- 一旦完成正式綁定，不應因畫面顯示編號變動而被動改寫。

## 結論
- 顯示可變，但 `id` 必須唯一且穩定。
- `label` 不得取代 `id`。
- `label` 也不得取代 `tableCode`。
- 內部資料識別以穩定 `id` 為基礎。
- 營運系統對接以 `tableCode` 為基礎。
- 顯示層則可自由使用 `label` 呈現。

## 第一次匯入的地圖識別原則
第一次匯入桌位資料時，應先以正式地圖 JSON 名稱或 `mapId` 確認目前匯入的是哪一張正式地圖。

此步驟的作用為：
- 確認匯入來源
- 確認使用者目前操作的是哪一張正式地圖
- 為後續建立該地圖的桌位綁定基準提供範圍依據

但需明確區分：
- 正式地圖 JSON 名稱或 `mapId`，只負責識別「哪一張地圖」
- 不足以作為每張桌位對接營運資料的完整唯一依據

## 第一次匯入的正式基準建立
第一次匯入時，不應先做差異比對，而應直接建立正式綁定基準。

建議流程：
- 使用者在 `table-admin` 選擇一張正式地圖
- 系統透過 JSON 名稱或 `mapId` 確認匯入來源
- 讀取該地圖中的桌位清單
- 為每一張桌位建立正式綁定資料
- 建立後作為後續重新匯入時的比對基準

補充原則：
- 第一次匯入時，重點是建立基準，不是比較差異。
- 第二次之後的重新匯入，才以既有綁定基準進行差異分析。

## 桌位管理責任再收斂
依目前討論結果，桌位管理模組的重點不再是維護場域結構，而是維護桌位營運設定。

因此桌位管理模組第一階段最重要的管理項目應為：
- 有多少座位
- 座位編號
- 最大訂單數 `maxActiveOrders`
- QR Code 是否已產生
- 對應網址
- 桌位是否停用

補充原則：
- 桌位管理模組不再承擔場域結構刪除責任。
- 桌位管理模組的核心是營運設定，而不是空間編輯。

## 桌位刪除責任調整
桌位刪除權力應移至餐廳地圖產生器。

原因如下：
- 桌位是否存在，本質上屬於地圖與場域結構問題。
- 若在桌位管理模組中直接刪除桌位，容易造成地圖結構與營運資料不同步。
- 由地圖產生器管理桌位刪除，較符合資料來源一致原則。

因此：
- `table-admin` 不提供桌位刪除作為主要操作。
- 桌位若不應再參與營運，第一階段以「停用」處理。
- 真正移除桌位，應回到地圖產生器修改正式地圖後，再由匯入流程反映。

## 停用取代刪除
為降低營運風險，桌位管理第一階段採：
- 停用
- 啟用

而非直接刪除。

這樣可保留：
- 舊桌號資料
- QR 對應紀錄
- 歷史訂單追溯能力

## 區域欄位語意調整
原本桌位管理中的區域概念，應由地圖本身承接。

原因如下：
- 區域、樓層、場域切分，已屬於正式地圖本身的結構資訊。
- 若桌位管理模組再維持另一套區域欄位，容易產生雙重來源與語意衝突。

因此：
- 原本 `area` / `area_name` 類概念，應逐步轉為由正式地圖負責。
- 桌位管理模組若仍需保留描述欄位，應改為地圖上的備註或對應顯示用途。
- 第一階段應避免讓 `table-admin` 再成為區域主資料來源。

## 第一階段桌位管理畫面重點
桌位管理畫面應收斂為營運管理清單，而非場域編輯工具。

建議顯示重點：
- 桌位編號
- 綁定狀態
- `maxActiveOrders`
- QR 狀態
- 入口網址
- 啟用 / 停用狀態
- 來自哪一張正式地圖

## 實作順序調整
依目前討論結果，桌位管理調整應先於櫃檯地圖整合模組實作。

原因如下：
- 櫃檯整合模組需要讀取已正式導入且完成綁定的桌位營運資料。
- 若桌位管理的導入與綁定機制尚未建立，櫃檯整合模組將缺乏穩定資料來源。
- 先做櫃檯整合，後續再重構桌位管理，會造成重複對接與額外返工。

因此實作順序應調整為：

### 第一階段：補強地圖桌位正式欄位
- 補齊正式地圖中的桌位欄位結構。
- 確保 `id`、`label`、`tableCode`、`maxActiveOrders` 等欄位可正式落地。
- 建立內部識別、顯示名稱與營運對接鍵的分層。

### 第二階段：重構 `table-admin` 為桌位營運管理中心
- 新增正式地圖清單與桌位導入能力。
- 建立第一次匯入的正式綁定基準。
- 建立重新匯入時的差異比對基礎。
- 管理 `tableCode`、`maxActiveOrders`、QR、網址、啟用停用等營運欄位。
- 將桌位刪除責任移回地圖產生器。

### 第三階段：完成正式地圖與桌位主資料橋接
- 確認地圖桌位與 `dinecore_tables` 的綁定穩定可用。
- 確認重新匯入不會破壞既有營運資料。
- 確認桌位營運主資料與正式地圖責任邊界已穩定。

### 第四階段：實作櫃檯地圖整合模組
- 套用正式地圖至櫃檯視圖。
- 顯示桌位狀態與訂單入口。
- 點擊桌位後進入既有訂單業務流程。
- 不重寫既有櫃檯核心模型。

## 順序結論
櫃檯整合模組屬於結果層。

在此之前，必須先完成：
- 地圖桌位正式結構
- 桌位管理的導入與綁定能力
- 正式地圖與營運桌位資料的穩定橋接

因此桌位管理調整應先做，櫃檯整合模組後做。

## 桌位主資料表策略
第一階段桌位導入與管理，優先從現有 `dinecore_tables` 出發。

原則如下：
- 不先新增過多新表，避免基礎結構過早複雜化。
- 先沿用既有 `dinecore_tables` 作為桌位營運主資料表。
- 視正式地圖導入需求，補充必要欄位。

此做法的目標為：
- 延續既有訂單、QR、session 與桌號管理流程。
- 降低重構範圍。
- 先建立穩定可用的正式導入機制。

## 欄位預留策略
依目前討論，桌位主資料欄位寧可先預留，也不要等功能啟用時才發現欄位不足。

因此：
- 若未來已知有高機率使用的欄位，應提前納入設計。
- 第一階段可不全部啟用，但不應完全缺失。
- 欄位存在不代表第一階段全部開放使用，仍須遵守責任邊界。

## 正式地圖桌位必須具備穩定唯一 ID
正式地圖中的每一張桌位，必須帶有穩定且唯一的桌位識別 ID。

此 ID 的用途為：
- 第一次匯入時建立正式綁定基準
- 後續重新匯入時進行比對
- 判斷哪一張桌位是同一張桌位
- 避免因顯示名稱、編號或排序變動造成錯誤對接

此 ID 必須符合以下原則：
- 建立後固定不變
- 不因排序、重排、刪除其他桌位而改變
- 不依賴 `label`
- 不依賴 `tableCode`
- 作為正式地圖桌位的內部唯一識別

## 重新匯入比對原則
重新匯入時，不得依賴以下欄位作為唯一比對依據：
- 顯示名稱
- 顯示編號
- `label`

原因如下：
- 上述欄位可能隨刪除、重排、重新命名而變動
- 若用這些欄位比對，將導致 QR、桌號與訂單資料錯位

因此重新匯入時應採用：
- 以正式地圖桌位穩定 `id` 為主比對鍵

## 資料表補欄位方向
若沿用 `dinecore_tables` 作為桌位主資料表，則應考慮補充可對應地圖來源的欄位。

建議至少評估：
- `map_id`
- `map_table_id`

欄位用途：
- `map_id`：表示此桌位目前來自哪一張正式地圖
- `map_table_id`：表示此桌位對應到正式地圖中的哪一張桌位

此設計可讓後續重新匯入時，能以穩定 ID 直接找到既有桌位主資料，而非透過名稱猜測。

## `tableCode` 不應直接綁定正式地圖 JSON 名稱
正式地圖 JSON 名稱或 `mapId` 的責任，是識別「哪一張地圖」，而不是識別「哪一張桌位」。

因此：
- `tableCode` 不應直接綁定 JSON 名稱。
- JSON 名稱不得作為桌位營運碼的唯一正式來源。

原因如下：
- 一張地圖內可能包含多張桌位，JSON 名稱只能識別地圖，無法完整識別桌位。
- 若地圖改名、替換或調整命名規則，將造成桌號、QR 與訂單資料不必要的連動。
- `tableCode` 若過度依附地圖名稱，會使營運桌號與地圖命名策略耦合過深。

## 建議做法
- JSON 名稱或 `mapId` 僅用於識別匯入來源。
- `tableCode` 應作為獨立營運桌號存在。
- 若需要提高匯入效率，可在第一次匯入時用地圖名稱作為 `tableCode` 的建議值來源之一。
- 但正式落地後，`tableCode` 不應因地圖 JSON 名稱變動而自動改寫。

## 結論
- 地圖識別屬於地圖層。
- `tableCode` 屬於營運桌號層。
- 兩者可以關聯，但不可直接綁死為同一語意來源。

## `tableCode` 產生方式
第一階段 `tableCode` 採自動產生，不提供使用者手動指定作為主要流程。

原因如下：
- `tableCode` 屬於系統營運對接鍵，不是主要使用者語意欄位。
- 若交由人工輸入，容易出現重複、拼字錯誤、命名不一致等問題。
- 使用者實際上更關心的是桌位顯示名稱、位置與狀態，而不是內部營運碼格式。
- 由程式自動產生，可確保唯一性、穩定性與後續對接一致性。

## 產生原則
- `tableCode` 由系統於第一次匯入時自動建立。
- 一旦建立完成，不應因 `label`、地圖名稱或排序變動而自動重編。
- `tableCode` 應視為正式營運鍵，供桌位管理、QR、訂單與 session 使用。

## 使用者可見性
- 使用者可在管理畫面中看到 `tableCode`，但不應依賴人工輸入建立。
- 若未來有特殊需求，可再討論是否提供高權限調整機制。
- 第一階段不以人工指定作為正常流程。

## 使用者語意應以業務行為為主
不論是一般使用者或管理者，都不需要理解資料庫如何設計，也不需要知道系統內部如何實作桌位綁定。

使用者真正關心的是：
- 這是哪一桌
- 這張訂單是哪一張
- 目前可做什麼業務動作

因此畫面語意應以業務行為為主，而非以資料庫鍵值為主。

## 使用者主要辨識欄位
第一階段使用者主要辨識資訊應為：
- `label`
- `orderNo`

欄位責任如下：
- `label`：讓使用者知道這是哪一桌
- `orderNo`：讓使用者知道這是哪一張訂單

## `tableCode` 的顯示定位
`tableCode` 雖然是系統內部重要的正式營運鍵，但不應作為使用者主要操作語意。

原則如下：
- 地圖上優先顯示 `label`
- 訂單列表與業務操作頁面優先顯示 `orderNo`
- `tableCode` 僅作為管理、綁定或除錯時的次要資訊
- 第一階段不應要求一般使用者依賴 `tableCode` 進行日常操作

## 結論
- 使用者要的是業務行為，不是資料庫結構。
- 管理者要的是可管理結果，不是內部實作細節。
- UI 應隱藏不必要的內部鍵值，只保留對業務有意義的資訊。

## `table-admin` 入口畫面原則
進入桌位管理模組後，第一步應先選擇地圖。

原因如下：
- 目前地圖本身已承接區域概念。
- 一張地圖可代表一個區域、樓層或場域範圍。
- 使用者在桌位管理時，應先確認自己正在管理哪一張地圖下的桌位。

因此第一階段入口流程應為：
- 先顯示正式地圖清單
- 使用者選擇一張地圖
- 再進入該地圖的桌位管理畫面

## 區域欄位語意調整補充
原本桌位管理中的區域欄位，因地圖已承接區域語意，應不再作為主欄位使用。

因此：
- 原本區域欄位的位置，可改為備註用途。
- 該欄位應用於補充桌位營運描述，而不是再次定義場域區域。

補充原則：
- 區域由地圖決定。
- 備註由桌位管理模組維護。
- 不應在桌位管理內再建立第二套區域主語意。

## 地圖匯入後的桌位管理畫面
當使用者選擇並匯入一張正式地圖後，畫面應直接顯示該地圖上所有桌位的清單列表，並同步生成所有桌位卡片。

此畫面的目標是：
- 讓使用者立即看到該地圖下有哪些桌位
- 直接管理每張桌位的營運設定
- 不再需要額外切換到其他桌位建立流程

## 桌位卡片保留與移除項目
### 應保留
桌位卡片應保留原本桌位管理中與營運直接相關的功能與欄位，包括：
- QR 產生器
- 入口網址
- 桌位相關操作按鈕
- `maxActiveOrders`
- 備註
- 啟用 / 停用
- 暫停點餐 / 凍結點餐功能

### 應移除
以下功能在新的桌位管理流程中意義不大，應移除：
- 上移
- 下移
- 刪除桌位按鈕

移除原因如下：
- 桌位排序主要由地圖本身與地圖清單決定，不再依靠桌位管理中的上下移動。
- 桌位是否存在屬於地圖結構責任，刪除權限應回到地圖產生器。

## 凍結功能保留原則
暫停點餐或凍結點餐功能必須保留。

原因如下：
- 這屬於桌位營運管理，而不是場域編輯。
- 現場可能需要暫時停止某桌位接單，但不代表要刪除或移除桌位。
- 此功能應仍由桌位管理模組負責。

## 桌位卡片欄位調整
依目前討論，桌位名稱不再由桌位管理模組編輯，而是直接沿用地圖產生器所產生的桌位名稱。

因此：
- 桌位名稱改為唯讀文字顯示
- 不提供桌位名稱輸入欄位
- 桌位名稱的修改責任回到地圖產生器

## 備註欄位
原本區域輸入欄位改為備註欄位。

用途為：
- 補充桌位營運說明
- 記錄特殊用途
- 記錄臨時提醒事項

此欄位不再承接區域語意。

## 桌位狀態顯示
桌位卡片中的狀態，應以營運狀態為主，而不是資料庫欄位原樣直出。

第一階段建議顯示以下三種狀態：
- `正常營運`
- `桌位暫停點餐`
- `已到目前線上最大訂單數`

狀態說明如下：
- `正常營運`：桌位可正常接單，且尚未達到同時有效訂單上限
- `桌位暫停點餐`：桌位因清潔、維修或其他原因暫時不可接單
- `已到目前線上最大訂單數`：桌位目前有效訂單數已達 `maxActiveOrders` 上限

補充原則：
- 狀態顯示以業務語意為主。
- 若同時存在多個內部條件，畫面應轉譯為對使用者最有意義的單一主狀態。
- 第一階段不直接暴露複雜的內部狀態組合給使用者。

## `maxActiveOrders` 設定
桌位卡片內需提供 `maxActiveOrders` 設定欄位。

用途為：
- 定義此桌位在未結帳前，同一時間最多允許存在的有效訂單數
- 作為桌位是否已達訂單上限的重要判斷依據

補充原則：
- 此欄位屬於桌位營運設定，應由桌位管理模組維護
- 此欄位為第一階段桌位卡片中的核心設定項目之一

## 桌位狀態判定來源
桌位卡片中的營運狀態，來源應以該桌實際送出且尚未結帳的訂單為準。

判定原則如下：
- 若桌位被設定為暫停點餐，主狀態顯示為 `桌位暫停點餐`
- 若桌位未暫停點餐，但未結帳訂單數已達 `maxActiveOrders`，主狀態顯示為 `已到目前線上最大訂單數`
- 若桌位未暫停點餐，且未結帳訂單數尚未達上限，主狀態顯示為 `正常營運`

## 未結帳訂單數的判定語意
第一階段中，桌位狀態所使用的「目前線上訂單數」，應理解為：
- 已送出
- 尚未結帳
- 仍屬於有效中的訂單

也就是說：
- 已送出且未結帳的訂單，會占用桌位可接單容量
- 若此數量已達 `maxActiveOrders`，則桌位不應再接受新的訂單

## 狀態優先序
為避免畫面同時顯示多種衝突語意，第一階段應採單一主狀態顯示。

優先序如下：
1. `桌位暫停點餐`
2. `已到目前線上最大訂單數`
3. `正常營運`

此規則可確保：
- 突發清潔、維修等人工凍結狀態，優先於一般訂單數量限制
- 畫面狀態對使用者維持單一明確語意

## 桌位卡片按鈕收斂
第一階段桌位卡片按鈕應只保留與營運直接相關的功能。

建議保留按鈕如下：
- `產生 / 重新產生 QR`
- `複製入口網址`
- `暫停點餐 / 恢復點餐`
- `儲存設定`
- `查看目前訂單`

## `查看目前訂單` 的用途
桌位卡片中的 `查看目前訂單` 應作為業務入口按鈕。

用途為：
- 讓管理者快速查看該桌目前的有效訂單
- 從桌位管理快速進入訂單相關內容
- 作為後續與櫃檯流程串接的入口之一

補充原則：
- 第一階段可先導向既有訂單列表或訂單詳情流程
- 不要求在桌位管理卡片內直接完成全部訂單操作
- 此按鈕屬於業務便利入口，應保留

## 桌位卡片清單排序
第一次匯入地圖後，以及後續桌位管理清單顯示時，桌位卡片應按照桌位編號順序呈現。

排序原則如下：
- 以使用者可理解的桌位編號順序為主
- 不再依賴原本桌位管理中的上移 / 下移排序機制
- 清單排序應與地圖桌位編號理解保持一致

補充原則：
- 若顯示名稱中包含可解析的編號，應依編號順序排列
- 若未來有特殊排序需求，再額外討論，但第一階段以編號順序為準

## Task B：`dinecore_tables` 欄位新增草案
第一階段 `Task B` 的目標，是在不破壞既有桌位、訂單、QR 與 session 流程的前提下，補足桌位匯入與營運設定所需欄位。

## 原則
- 只新增欄位，不改舊欄位語意
- 不移除既有欄位
- 不重命名既有欄位
- 不在第一步改動舊 API 主要流程
- 以安全預設值與可為空欄位為主，降低舊功能風險

## 建議新增欄位
### `map_id`
用途：
- 表示此桌位目前對應哪一張正式地圖

建議：
- 型別：`VARCHAR(...) NULL`
- 第一階段可允許為空

### `map_table_id`
用途：
- 表示此桌位目前對應正式地圖中的哪一張桌位
- 作為重新匯入時的重要比對依據之一

建議：
- 型別：`VARCHAR(...) NULL`
- 第一階段可允許為空

### `max_active_orders`
用途：
- 表示此桌位在未結帳前同時允許存在的有效訂單數上限

建議：
- 型別：`INT NOT NULL`
- 預設值：`1`

### `note`
用途：
- 作為桌位營運備註
- 取代原本桌位管理中區域輸入欄位的主要使用方向

建議：
- 型別：`TEXT NULL`

## 第一階段不建議立即加入的高風險限制
以下限制第一階段不建議急著加入，以免在匯入流程尚未完全定稿前卡住既有資料：
- 過早加入複合唯一限制
- 對 `map_id` / `map_table_id` 施加過嚴的 not null 限制
- 對既有桌位資料立即強制補齊所有新欄位

## 第一階段預期效果
完成此欄位新增後，應可達成：
- 舊有桌位功能持續可用
- 舊有訂單流程不受影響
- 後續正式地圖匯入可有落地欄位可用
- `table-admin` 可逐步開始承接新桌位營運設定

## `map_id` 與 `map_table_id` 格式定案
為了降低維護成本並讓匯入流程保持直接可理解，第一階段 `map_id` 與 `map_table_id` 採用最直接方式處理。

定案如下：
- `map_id` 直接沿用正式地圖 JSON 中的 `map.id`
- `map_table_id` 直接沿用正式地圖桌位資料中的 `id`

## 採用原因
- 不需額外轉碼或重新產生另一套對接鍵
- 第一次匯入可直接落地
- 後續重新匯入可直接比對
- 除錯與維護時可直接對照正式地圖資料
- 可降低資料流複雜度

## 第一階段原則
- 不另建額外抽象鍵
- 不對 `map_id` / `map_table_id` 做額外加密或特殊封裝
- 以好維護、可追溯、易理解為優先

## Task C：正式地圖匯入 API 最小動作
第一階段正式地圖匯入 API，先收斂為四個最小必要動作，不過度拆分。

### 1. 取得正式地圖清單
用途：
- 提供 `table-admin` 入口選擇可匯入的正式地圖

建議至少回傳：
- `mapId`
- `name`
- `updatedAt`
- 可選：桌位數量

### 2. 取得單張正式地圖內容
用途：
- 讓 `table-admin` 讀取使用者所選正式地圖的桌位資料

建議至少回傳：
- `map`
- `tables`

補充原則：
- 僅讀正式地圖
- 不讀草稿資料

### 3. 第一次匯入桌位
用途：
- 將正式地圖中的桌位資料寫入 `dinecore_tables`
- 建立 `map_id` / `map_table_id`
- 自動產生 `tableCode`
- 建立 `max_active_orders` 預設值
- 建立正式綁定基準

### 4. 重新匯入桌位
用途：
- 依 `map_table_id` 比對既有桌位
- 更新地圖層相關資料
- 保留既有營運設定
- 不直接刪除舊桌位資料
- 對已從地圖消失的桌位標記為待處理或停用候選

## 第一階段 API 收斂原則
- 不將匯入 API 拆得過細
- 先完成能支撐 `table-admin` 主流程的最小能力
- 後續若出現更細緻需求，再另行擴充

## `tableCode` 第一階段生成格式
為了確保不出錯、好維護、且不受地圖命名與顯示編號影響，第一階段 `tableCode` 採全域流水號自動生成。

建議格式：
- `T0001`
- `T0002`
- `T0003`

## 生成原則
- 由系統自動產生
- 採固定前綴，例如 `T`
- 數字補零，方便排序與閱讀
- 一旦建立完成，不因地圖名稱、`label`、排序或重新匯入而重編
- 以全域唯一為原則

## 不採用的方式
第一階段不採用以下來源生成 `tableCode`：
- 地圖名稱
- JSON 檔名
- 區域名稱
- 顯示編號 `label`
- 重新排序後的桌位順序

原因如下：
- 上述資料皆可能變動
- 若直接拿來生成正式營運鍵，容易造成 QR、訂單與桌位綁定錯亂

## 建議實作方式
- 匯入時先查詢目前資料庫中既有最大序號
- 下一筆以遞增方式產生新的 `tableCode`
- 建立後即視為正式營運桌號，不再自動重編

## `label` 與備註的責任分工補充
桌位 `label` 應直接沿用正式地圖資料，不在 `table-admin` 內重新產生，也不作為日常編輯欄位。

原因如下：
- `label` 屬於地圖桌位顯示名稱的一部分
- 其來源應維持單一，避免地圖與桌位管理出現雙重命名來源
- 若在 `table-admin` 中再次修改，容易造成地圖與營運畫面顯示不一致

若需要提供使用者自訂或現場營運說明，應改由備註欄位承接。

備註欄位可用於：
- 給現場人員看的補充說明
- 標示特殊用途
- 顯示暫時性提醒
- 承接不適合寫回正式地圖名稱的文字

## 結論
- `label`：沿用正式地圖資料，唯讀顯示
- `note`：提供使用者自訂與營運補充說明
- 不在 `table-admin` 內建立第二套桌位名稱編輯流程

## 正式地圖與桌位主資料的完整關聯原則
目前討論的方向，不是要將正式地圖與桌位主資料完全切開，而是要做到：
- 責任分層
- 關聯完整

也就是說，兩者應互相引用、可互相追溯，但不能互相取代責任。

## 正式地圖負責的內容
正式地圖應負責：
- 地圖本身
- 桌位位置
- 桌位顯示名稱 `label`
- 桌位穩定唯一 ID
- 桌位在場域中的視覺與結構資訊

## 桌位主資料負責的內容
桌位主資料應負責：
- `tableCode`
- `maxActiveOrders`
- QR
- 入口網址
- 暫停點餐 / 恢復點餐
- 備註
- 啟用 / 停用等營運設定

## 關聯方式
為了讓正式地圖與桌位資料完整關聯，建議採雙向可追溯方式：

### 正式地圖桌位
可承接：
- `id`
- `label`
- `tableCode`

### `dinecore_tables`
可承接：
- `code`
- `map_id`
- `map_table_id`

如此可達成：
- 從地圖桌位找到桌位主資料
- 從桌位主資料回推其來源地圖與桌位
- 重新匯入時以穩定關聯做比對
- 櫃檯整合時同時取得位置與營運設定

## 結論
- 不是完全切開
- 而是完整關聯、責任分層
- 正式地圖不取代桌位主資料
- 桌位主資料也不反向接管地圖結構

## `tableCode` 來源更新定案
經進一步討論後，第一階段 `tableCode` 的來源改由正式地圖承擔。

新版定案如下：
- `tableCode` 由 `restaurant-map-editor` 在正式地圖資料中產生並保存
- `table-admin` 匯入時直接沿用正式地圖中的 `tableCode`
- `dinecore_tables.code` 應與正式地圖桌位中的 `tableCode` 對應一致

## 採用原因
- 可讓 `tableCode` 的來源單一
- 正式地圖與桌位主資料可維持天然一致
- 匯入流程更像正式資料同步，而不是匯入後再補產營運鍵
- 櫃檯整合模組在讀取正式地圖時，也可直接取得完整對接鍵

## 責任分工更新
### `restaurant-map-editor`
- 負責產生正式地圖桌位資料中的 `tableCode`
- 確保每張正式桌位都具有穩定且可沿用的 `tableCode`

### `table-admin`
- 匯入時沿用正式地圖中的 `tableCode`
- 不再作為 `tableCode` 的主來源生成位置
- 主要負責營運設定與桌位主資料承接

## 更新後原則
- `tableCode` 一旦建立，不因 `label`、排序、刪除其他桌位或地圖重排而重編
- 匯入流程應以正式地圖內既有 `tableCode` 為準
- 若未來文件中仍出現「第一次匯入時由 `table-admin` 生成 `tableCode`」的舊說法，應以本段新版定案為準

## `tableCode` 產生時機定案
`tableCode` 不要求在桌位建立當下立即生成。

第一階段定案為：
- 使用者按下正式儲存按鈕時
- 系統統一檢查正式地圖中的桌位資料
- 若有缺少 `tableCode` 的桌位，於此時補齊
- 同時檢查整張正式地圖中的 `tableCode` 唯一性

## 採用原因
- 編輯過程中不必過早承擔營運鍵生成責任
- 正式儲存時才是正式資料落地時機，責任最清楚
- 可在單一時機統一處理補齊與驗證
- 可避免編輯途中反覆生成又重整的複雜情況

## 正式儲存時的要求
按下正式儲存時，系統應保證：
- 每張桌位都有 `id`
- 每張桌位都有 `label`
- 每張桌位都有 `tableCode`
- `tableCode` 在同一張正式地圖內不得重複
- 若系統設計採全域唯一，則也需同時檢查跨地圖重複

## 結論
- `tableCode` 的補齊時機：正式儲存時
- `tableCode` 的唯一性驗證時機：正式儲存時
- 正式地圖一旦儲存完成，即應具備完整可匯入的桌位正式資料

## 正式儲存時的自動修正原則
正式儲存時，若發現正式地圖桌位資料存在缺漏或系統鍵衝突，應由系統自動補齊與自動修正，不要求使用者自行排除。

原因如下：
- 使用者不需要理解內部資料鍵的生成與驗證規則
- 將系統資料完整性責任交給使用者，容易造成操作負擔與錯誤
- `tableCode`、內部唯一 ID 等欄位屬於系統責任，應由系統處理

## 自動修正適用範圍
正式儲存時，系統應可自動處理：
- 缺少 `tableCode`
- `tableCode` 衝突
- 缺少必要系統欄位
- 系統預設值補齊

## 自動修正邊界
自動修正原則應優先處理系統欄位，不應隨意改動使用者可感知的顯示資料。

可自動修正：
- `tableCode`
- 穩定唯一 `id`
- `maxActiveOrders` 預設值補齊

不應任意自動改動：
- `label`
- 幾何位置
- 備註

## 結論
- 正式儲存時，系統負責補齊與修正正式桌位資料
- 使用者不需理解如何修正系統鍵值
- 使用者主要負責編輯地圖內容，系統負責保證正式資料完整性

## 定案摘要

### 一、整體方向
- 不直接修改既有已完成模組的核心邏輯。
- 先補強地圖桌位正式資料與 `table-admin`，再做櫃檯地圖整合模組。
- 正式地圖與桌位主資料不是切開，而是完整關聯、責任分層。

### 二、責任分工
- `restaurant-map-editor`：負責正式地圖、桌位位置、桌位顯示名稱、桌位穩定 ID、正式桌位 `tableCode`。
- `table-admin`：負責桌位營運主資料、QR、入口網址、`maxActiveOrders`、備註、凍結點餐、啟用停用、查看目前訂單。
- 櫃檯地圖整合模組：最後才做，負責套用正式地圖並接入既有業務流程。

### 三、正式地圖桌位資料
正式地圖桌位應至少包含：
- `id`
- `label`
- `tableCode`
- `maxActiveOrders`
- 幾何位置資料

原則：
- `id` 為穩定唯一識別，不可因重排或改名改變。
- `label` 為顯示名稱，可變。
- `tableCode` 為正式營運對接鍵。

### 四、`tableCode` 定案
- `tableCode` 來源由正式地圖承擔。
- 在按下正式儲存時統一補齊與驗唯一。
- `tableCode` 採全域流水號格式，例如：`T0001`、`T0002`。
- 一旦建立完成，不因 `label`、排序或其他桌位刪除而重編。

### 五、正式儲存原則
正式儲存時，系統應自動補齊與修正正式桌位資料：
- 補齊 `tableCode`
- 驗證唯一性
- 補齊必要系統欄位與預設值

使用者不需要理解系統鍵如何修正。

### 六、`dinecore_tables` 方向
第一階段沿用現有 `dinecore_tables`，只新增欄位，不改舊欄位語意。

建議補充：
- `map_id`
- `map_table_id`
- `max_active_orders`
- `note`

格式定案：
- `map_id = 正式地圖 map.id`
- `map_table_id = 正式地圖桌位 id`

### 七、桌位管理主流程
- 進入 `table-admin` 後先選正式地圖。
- 匯入後直接顯示該地圖所有桌位卡片。
- 第一次匯入直接建立正式綁定基準。
- 重新匯入時依穩定 ID 比對，不靠名稱或編號。
- 正式地圖不自動寫入資料庫，匯入主動權在使用者。

### 八、桌位卡片定案
保留欄位與功能：
- 桌位名稱 `label`（唯讀）
- 備註 `note`
- `maxActiveOrders`
- QR 狀態
- 入口網址
- 暫停點餐 / 恢復點餐
- 查看目前訂單
- 儲存設定

移除：
- 上移
- 下移
- 刪除桌位

### 九、桌位狀態定案
第一階段主狀態只有三種：
- `正常營運`
- `桌位暫停點餐`
- `已到目前線上最大訂單數`

判定來源：
- 以該桌已送出且尚未結帳的有效訂單數為準

優先序：
1. `桌位暫停點餐`
2. `已到目前線上最大訂單數`
3. `正常營運`

### 十、使用者語意
- 使用者主要看 `label` 與 `orderNo`
- `tableCode` 屬於系統內部正式營運鍵，不作為主要操作語意
- 若要提供自訂說明，使用備註欄位承接

6. 桌位狀態與訂單入口整合
7. 櫃檯地圖整合模組

## 總結原則
- 先補資料，再補 API，再改畫面，最後做整合模組
- 先穩定正式資料來源與桌位主資料關聯，再談地圖式櫃檯
- 不讓櫃檯整合模組反過來承擔建模責任
