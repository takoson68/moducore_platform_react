# 餐廳地圖產生器

## 文件定位
本文件整理 DineCore 餐廳地圖產生器第一版（MVP）的功能範圍與操作流程。

目標是先建立一套可建立、編輯、儲存與重新載入的地圖編輯系統，作為後續桌位管理、櫃台營運與場域視覺化的基礎。

補充原則：

- 第一版先完成餐廳地圖產生。
- 下一階段桌位編輯仍放在同一個產生器內，不另開獨立頁面。
- 同一個產生器內分為兩個工作模式：
  - `Map Edit Mode`
  - `Table Edit Mode`
- 本系統採用 `SVG + HTML` 結構。
- `SVG` 負責地圖與桌位圖形顯示。
- `HTML` 負責操作 UI 與疊加控制。
- 本文件中的 `Workspace` 不等同於 HTML `<canvas>` 元素。
- 餐廳地圖產生器應以新模組方式開發，不直接在既有桌位管理模組上擴寫。
- 既有桌位管理模組在此階段先不修改，僅保留未來資料庫對接需求。
- 新模組功能完成並驗證完整後，再以淘汰方式移除舊桌位管理模組。
- 在舊模組中直接實作新模組功能，容易造成資料共用邊界混亂，因此目前應避免此做法。
- 新模組與舊模組的資料結構應先維持分流，避免開發期間打亂既有結構。

## 1. 地圖基本資料
每張地圖至少需要以下基本屬性：

- 地圖名稱 `name`
- 地圖寬度 `width`
- 地圖高度 `height`

規則如下：

- 以上屬性應可隨時修改。
- 地圖尺寸需與設定值即時綁定。
- 地圖寬度與高度應為大於 `0` 的有效數值。
- 若名稱或尺寸未填完整，系統不應允許建立地圖。
- 第一版不處理底圖匯入。
- 場域結構直接由編輯器內的繪圖工具建立。

## 2. 建立地圖流程
建立新地圖時，使用者流程如下：

1. 輸入地圖名稱。
2. 設定地圖尺寸。
3. 建立地圖。
4. 系統自動進入編輯模式 `Edit Mode`。

建立完成後，使用者即可直接開始繪製地圖內容。

補充流程：

- 進入餐廳地圖產生器頁面後，應提供明確的「新增地圖」按鈕。
- 使用者點擊後，系統顯示建立地圖表單，讓使用者輸入地圖名稱與地圖尺寸。
- 使用者確認送出後，系統建立新地圖並產生空白地圖。
- 新建立的空白地圖第一版應顯示網格線，作為繪製參考。
- 建立完成後，系統預設進入 `Edit Mode` 下的 `Map Edit Mode`。
- 系統若存在多張地圖，需明確維護目前正在查看或編輯的 `active map`。
- 建立新地圖後，新地圖自動成為目前的 `active map`。
- 第一版同一時間只允許一張 `active map` 進入 `Edit Mode`。
- 第一版應提供地圖清單或等效切換入口，讓使用者能切換 `active map`。
- 若 `active map` 被刪除，系統應自動切換到下一張可用地圖；若已無地圖，則回到空狀態。
- 若使用者刪除目前的 `active map`，且該地圖有未儲存變更，系統應先提示。
- 確認刪除後，系統才移除該 `map`。
- 移除 `active map` 後，相關的 `active object`、`draftState`、`working mode` 也應一併清理。

## 3. 編輯介面結構
第一版編輯介面分為兩個主要區塊：

- 左側：
  - 網站導覽列
  - 負責系統頁面切換。
- 右側：
  - 工作區域 `Workspace`
  - `Workspace` 包含：
    - 編輯工具列 `Toolbar`
    - 編輯操作面 `Editor Surface`
      - `SVG Root` 繪圖根節點
      - `Map Layer`
      - `Table Layer`
      - `UI Layer / Overlay Layer`

介面原則如下：

- 工具列位於 `Editor Surface` 上方。
- `Editor Surface` 是主要操作區域。
- 地圖圖形與桌位圖形皆顯示於 `SVG Root` 內。
- `Map Layer` 負責地圖圖形顯示。
- `Table Layer` 負責桌位圖形顯示。
- 操作控制與疊加 UI 由 `UI Layer / Overlay Layer` 負責。
- 第一版 `UI Layer / Overlay Layer` 至少承載模式切換、建立地圖、儲存、完成編輯與刪除等操作 UI。
- `UI Layer / Overlay Layer` 負責互動控制，不直接承載地圖圖形資料。
- 第一版 `SVG Root` 內部圖層順序應固定為：`Map Layer` 在下層，`Table Layer` 在上層。
- `UI Layer / Overlay Layer` 為 HTML 疊加層，顯示於 `SVG Root` 之上。
- 若同一位置同時存在地圖圖形與桌位圖形，視覺上以 `Table Layer` 優先顯示。
- 命中與操作仍需遵守當前工作模式邊界，不因圖層顯示順序改變模式規則。
- 除了繪圖工具外，頁面也應提供一般功能按鈕，例如建立新地圖、儲存地圖與模式切換。
- 使用者在進入 `Map Edit Mode` 後，應可直接從上方 UI 看見地圖繪製相關工具。
- 使用者啟用 `Polyline` 後，即可直接在 `SVG Root` 上開始繪製折線。
- 新建立地圖或重新載入地圖後，第一版應以穩定且可立即操作的初始視圖顯示地圖。
- 第一版可採用預設縮放倍率顯示，並讓地圖主要區域落在可視範圍內。

## 4. 工具列設計
第一版工具列提供以下工具：

- `Polyline`
- `Rectangle`
- `Circle`
- `Text`

使用規則：

- 同一時間只允許一種工具處於啟用狀態。
- 工具僅用於新增地圖元素。
- 本系統不提供 `Select` 工具。
- 物件操作將透過單物件編輯模式 `Object Edit Mode` 完成。

補充規則：

- 在 `Edit Mode` 下，若某繪圖工具已啟用，點擊空白 `Editor Surface` 代表開始建立該工具的新物件。
- 在 `Edit Mode` 下，若目前沒有任何繪圖工具啟用，點擊空白 `Editor Surface` 不產生新物件。
- 第一版工具列僅服務 `Map Edit Mode`。
- 後續若切換到 `Table Edit Mode`，工具列將改為桌位相關工具，不與地圖繪圖工具混用。
- `Map Edit Mode` 與 `Table Edit Mode` 的切換入口，應設置於 `Workspace` 上方的模式切換區。
- 第一版建議使用明確的 `tab` 或 `segmented control` 呈現，不使用隱藏式切換。
- 模式切換 UI 應獨立於繪圖工具列，避免與工具啟用狀態混淆。
- 進入 `Map Edit Mode` 後，第一版預設不自動啟用任何繪圖工具。
- 使用者需明確點擊工具按鈕後，才開始建立對應物件。

## 5. 物件編輯模式（Object Edit Mode）
當使用者在 `Edit Mode` 中點擊 `SVG Root` 內的某個物件時，系統會進入該物件的編輯模式。

進入編輯模式後：

- 該物件顯示編輯控制，例如節點。
- 其他所有物件半透明顯示。
- 其他物件不可操作。
- `Editor Surface` 只允許編輯當前物件。

此模式可避免同時誤操作多個物件，並保持操作焦點。

補充規則：

- `Edit Mode` 為地圖層級編輯模式。
- `Object Edit Mode` 為 `Edit Mode` 內的單物件聚焦編輯狀態。
- `Object Edit Mode` 不是獨立主模式。
- `Object Edit Mode` 是 `Map Edit Mode` 或 `Table Edit Mode` 底下的單物件聚焦編輯狀態。
- 在 `View Mode` 下，點擊物件只作為查看，不進入物件編輯模式。
- 進入 `Object Edit Mode` 後，工具列暫時鎖定。
- 在 `Object Edit Mode` 下，不允許切換其他繪圖工具。
- 上述限制會持續到退出該物件編輯模式為止。
- 在 `Object Edit Mode` 下，點擊空白 `Editor Surface` 不會退出編輯模式。
- 退出 `Object Edit Mode` 僅能透過 `Esc` 或「完成編輯」按鈕。
- 進入 `Map Edit Mode` 或 `Table Edit Mode` 後，`Object Edit Mode` 才可能被啟動。

## 6. 折線工具 `Polyline`
折線工具是地圖繪製的核心功能，主要用於畫出：

- 餐廳外牆
- 隔間
- 區域邊界
- 動線邊界

### 6.1 操作流程
1. 點擊 `SVG Root` 新增第一個節點。
2. 持續點擊新增後續節點。
3. 系統依節點順序自動生成線段。
4. 使用雙擊或 `Enter` 結束繪製。

補充規則：

- 第一版 `Polyline` 至少需有兩個節點才可成立。
- 若節點數不足即結束繪製，系統不建立正式物件。

### 6.2 編輯折線
當使用者點擊折線物件時：

- 系統進入折線編輯模式。
- 顯示所有節點。
- 可拖曳節點調整形狀。
- 可刪除節點。
- 可刪除整條折線物件。
- 可新增節點。

同時：

- 其他物件半透明。
- 其他物件不可操作。

補充規則：

- 新增節點方式為點擊既有線段插入節點。
- 第一版不支援拖曳尾端延伸線段。
- 第一版不支援拆分 `polyline`。
- 第一版支援將單一線段由直線轉為簡單曲線段。
- 曲線段僅採用 `Quadratic Bezier`。
- 每一段 segment 僅允許為 `line` 或 `quadratic`。
- 曲線控制僅使用單一控制點，不提供雙控制柄。
- 曲線功能僅在 `Polyline` 的 `Object Edit Mode` 中提供。
- 當滑鼠 `hover` 某一段 segment 時，系統僅對該段顯示一個空心控制點。
- 未被 `hover` 的 segment 不顯示空心控制點。
- 直線段的空心控制點預設顯示於該段幾何中點。
- 拖曳空心控制點後，該段由 `line` 轉為 `quadratic`，拖曳位置成為控制點。
- 若該段已為 `quadratic`，再次拖曳同一空心控制點可調整曲率。
- 第一版應提供明確操作，讓曲線段可還原為直線段。
- 正在繪製 `Polyline` 時，點擊既有物件不進入物件編輯模式。
- 正在繪製 `Polyline` 時，`Esc` 應優先作為取消當前繪製行為。
- 正在繪製 `Polyline` 且尚未完成時，不允許切換其他工具。
- 進入 `Polyline` 的 `Object Edit Mode` 後，應支援刪除整條折線物件。
- 刪除整條折線後，系統應退出該物件的 `Object Edit Mode`。

### 6.3 退出編輯模式
可透過以下方式退出：

- 點擊「完成編輯」按鈕。
- 按 `Esc`。

退出後：

- 所有物件恢復正常顯示。
- 系統回到一般操作狀態。

### 6.4 設計原則
第一版折線工具維持輕量化，不提供過度複雜的路徑能力。

第一版只處理：

- 節點新增
- 節點拖曳
- 節點刪除
- 單段曲線控制點拖曳
- 曲線段還原直線
- 水平吸附
- 垂直吸附

第一版不處理：

- `Cubic Bezier`
- 複雜路徑布林運算
- 進階向量編修功能

補充規則：

- 第一版只有 `polyline` 支援節點編輯。
- 第一版只有 `polyline` 支援單段曲線編輯。

## 7. 磁點吸附功能
為了更容易畫出直角場域結構，折線工具需提供磁點吸附功能。

補充規則：

- 第一版空白地圖顯示的 `grid` 僅作為視覺參考。
- 第一版 `grid` 不參與自動吸附。
- 第一版吸附邏輯仍以 `polyline` 的水平 / 垂直 / 起點吸附為主。

### 7.1 前一節點吸附
新增節點時，系統以前一個節點作為基準：

- 若滑鼠水平偏差較小，吸附為水平線。
- 若滑鼠垂直偏差較小，吸附為垂直線。

此機制可快速畫出：

- 直角牆面
- 走道邊界
- 矩形區域輪廓

### 7.2 起點吸附
在繪製最後一段線時，系統除參考前一節點外，也會參考起始節點位置。

若滑鼠接近起點的 `x` 或 `y` 軸位置，系統自動吸附，使最後一段線更容易與起點對齊。

此機制有助於：

- 收尾形成整齊外框
- 建立更準確的矩形或封閉區域
- 降低手動對齊成本

## 8. 矩形與圓形工具
矩形與圓形工具用於繪製簡單區域，例如：

- 包廂區
- 吧台
- 廚房
- 區域框

### 8.1 操作方式
1. 在 `Editor Surface` 上點擊並拖曳。
2. 生成圖形。
3. 放開滑鼠完成。

補充規則：

- 第一版 `rect` 與 `circle` 需達到基本拖曳距離後才建立物件。
- 若僅點擊未形成有效尺寸，系統不建立正式物件。

### 8.2 編輯圖形
當使用者點擊圖形時：

- 系統進入該圖形的物件編輯模式。
- 可操作：
  - 移動
  - 調整大小
  - 刪除

同時：

- 其他物件半透明。
- 其他物件不可操作。

補充規則：

- `rect`、`circle` 僅支援移動、調整大小、刪除。
- `rect`、`circle` 不支援節點級編輯。
- 第一版應支援刪除整個 `rect` 或 `circle` 物件。
- 刪除圖形後，系統應退出該物件的 `Object Edit Mode`。

## 9. 文字工具
文字工具用於標示區域名稱，例如：

- 內用區
- 戶外區
- 包廂
- 吧台

第一版需支援：

- 新增文字
- 編輯文字內容
- 拖曳文字位置

補充流程：

- 使用者啟用 `Text` 工具後，可於 `Editor Surface` 上建立文字物件。
- 建立文字後，系統應提供輸入或編輯文字內容的操作入口。
- 若使用者未輸入有效文字內容，第一版不應建立空白文字物件。

文字被點擊時也會進入物件編輯模式。

補充規則：

- `text` 僅支援移動、調整大小、刪除。
- `text` 不支援節點級編輯。
- 第一版 `text` 的調整大小以文字框範圍為主，不延伸為完整排版系統。
- 第一版 `text resize` 不影響字級，只調整文字框範圍。
- 第一版文字不處理自動換行排版規則。
- 文字命中區域以文字框 `bounding box` 為準。
- 第一版應支援刪除整個 `text` 物件。
- 刪除文字後，系統應退出該物件的 `Object Edit Mode`。

## 10. 地圖儲存與重新編輯
完成繪製後，系統需能儲存完整地圖資料，包括：

- 地圖基本資料
- 所有繪圖物件
- 各物件幾何資料、節點與座標

系統需支援以下能力：

- 重新載入地圖
- 再次進入編輯模式
- 在既有地圖上持續修改

補充規則：

- 使用者在繪製過程中或完成後，皆可主動執行儲存。
- 第一版應支援草稿存檔，可保存尚未完成的繪製狀態。
- 草稿存檔時，系統除正式資料外，也應一併保存目前尚未完成的編輯中狀態。
- 重新載入草稿後，系統應可還原到先前的編輯狀態，讓使用者繼續繪製。
- 若目前 `active map` 存在未儲存變更，使用者切換地圖、離開頁面或刪除地圖前，系統應先提示。
- 提示選項至少包含：`儲存後繼續`、`不儲存直接離開`、`取消本次操作`。
- 若目前存在尚未完成的繪製、`Object Edit Mode` 或其他編輯中狀態，仍應先完成、取消或退出，再處理切換。

### 10.1 模式區分
系統需區分兩種模式：

- `View Mode`
  - 用於：
    - 櫃台桌況顯示
    - 場域可視化
  - 不允許編輯。
- `Edit Mode`
  - 用於：
    - 地圖繪製
    - 地圖修改
    - 物件編輯
  - 內部再細分為：
    - `Map Edit Mode`：負責場域圖形與文字編輯
    - `Table Edit Mode`：負責桌位新增、移動、編輯與刪除

補充規則：

- 地圖載入後預設為 `View Mode`。
- 只有當使用者主動進入 `Edit Mode`，才允許修改地圖。
- 使用者由 `Edit Mode` 進入實際工作時，需明確選擇 `Map Edit Mode` 或 `Table Edit Mode`。
- `Map Edit Mode` 與 `Table Edit Mode` 之間的切換，必須先退出當前的 `Object Edit Mode`。
- 若目前存在尚未完成的繪製行為，應先完成或取消後，才允許切換工作模式。
- 使用者在 `Map Edit Mode` 與 `Table Edit Mode` 間切換時，系統應清除當前 `active object` 或 `active table`。
- 模式切換後，不保留上一模式的 `Object Edit Mode` 狀態。
- 切換到另一模式後，命中判定與操作邏輯應立即切換到對應資料集合。

### 10.2 編輯狀態資料分工
為了避免互動邏輯混亂，第一版需明確區分「主資料」與「編輯中暫存狀態」。

- 主資料包含：`Map`、`objects`、`tables`。
- 編輯中暫存狀態包含：目前工作模式、啟用中的工具、目前選中的物件、尚未完成的繪製資料。
- 尚未完成的 `Polyline` 不應直接寫入正式 `objects`，應先存在編輯中暫存狀態。
- 只有在使用者完成建立後，該物件才寫入正式資料集合。
- 取消繪製時，應只清除編輯中暫存狀態，不影響已存在的正式資料。
- 若使用者執行草稿存檔，未完成的繪製資料可隨編輯中暫存狀態一起被保存。

### 10.3 儲存資料落地原則
第一版儲存時，應以可重新載入與可重新編輯為優先。

- 儲存內容應以世界座標資料為準，不儲存畫面縮放後座標。
- 儲存資料應包含地圖主資料與已完成的地圖物件資料。
- 若已進入桌位編輯實作階段，儲存資料可再包含 `tables` 集合。
- 若本次為草稿存檔，則可額外包含尚未完成的繪製中資料與編輯中狀態。
- 重新載入後，系統應可還原到可繼續編輯的資料狀態，而不是僅還原畫面結果。
- 儲存成功後，系統應將目前 `active map` 的未儲存狀態清除。
- 第一版可區分「草稿儲存」與「正式儲存」的操作語意，但兩者皆以同一套可重新編輯資料為基礎。
- 第一版不輸出圖片檔，儲存結果以可再載入的地圖資料為主。

第一版地圖產生器收斂為以下能力：

- 建立地圖
- 修改地圖名稱與尺寸
- 折線繪製
- 折線節點編輯
- 水平與垂直吸附
- 矩形繪製
- 圓形繪製

## 11.1 Map Edit Mode 最低互動規則

為了讓地圖編輯可直接進入實作，第一版需定義最小互動流程。

### 1. 啟用地圖工具

- 使用者需先進入 `Map Edit Mode`。
- 第一版 `Toolbar` 僅提供地圖工具：`Polyline`、`Rectangle`、`Circle`、`Text`。
- 同一時間只允許一種地圖工具處於啟用狀態。

### 2. 點擊空白編輯區域

- 若已有啟用的地圖工具，點擊空白 `Editor Surface` 代表開始建立新地圖物件。
- 若目前沒有任何工具啟用，點擊空白 `Editor Surface` 不建立新物件。
- 在地圖物件的 `Object Edit Mode` 下，點擊空白 `Editor Surface` 不會退出編輯模式。

### 3. 點擊地圖物件

- 在 `Map Edit Mode` 下點擊地圖物件，可進入該物件的 `Object Edit Mode`。
- 同一時間只允許一個地圖物件進入編輯。
- 命中判定只針對 `objects`。
- 若多個地圖物件重疊，命中判定優先以較上層物件為準。

### 4. 物件編輯範圍

- `polyline` 支援節點層級編輯。
- `rect`、`circle`、`text` 僅支援物件層級編輯。
- `rect`、`circle`、`text` 不支援節點級編輯。

### 5. 繪製中狀態

- 正在繪製 `Polyline` 或建立其他新圖形時，不進入既有物件的 `Object Edit Mode`。
- 若目前存在尚未完成的繪製行為，應先完成或取消後，才允許切換工具或工作模式。
- `Esc` 在繪製中狀態下，應優先取消當前繪製行為。

### 6. 模式限制

- 在 `Map Edit Mode` 下，不允許建立或編輯 `tables`。
- 在地圖物件的 `Object Edit Mode` 下，`Toolbar` 鎖定。
- 在地圖物件的 `Object Edit Mode` 下，不允許切換其他地圖工具。
## 12. 下一階段銜接
完成以上 MVP 後，再進入下一階段：

- 在地圖上加入桌位模型
- 桌位與營運資料串接
- 接入櫃台營運系統
- 實現桌況視覺化管理
- 後續提供櫃台功能模組匯入地圖草稿或完成地圖，作為定位、訂餐與營運管理畫面基礎

補充開發原則：

- 第一階段先完成 `Map Edit Mode`，確保地圖繪製、物件編輯、儲存與重新載入可穩定運作。
- 第二階段再接入 `Table Edit Mode`，處理桌位圖層、桌位互動與桌位資料分流。
- 最後階段才處理與既有資料庫的對接與舊桌位管理模組替換。

補充規則：

- 桌位編輯不是獨立產生器，而是同一個餐廳地圖產生器中的第二工作階段。
- 畫面層先維持同一個 `Workspace`。
- 邏輯與資料結構上需與地圖物件分流。
- 餐廳地圖產生器應作為新的功能模組獨立開發。
- 既有桌位管理模組在新模組完成前不承接地圖產生器功能。
- 新模組的目標是最終與既有資料庫完成對接，而不是與舊模組共用開發流程。
- 後續櫃台功能模組應能匯入餐廳地圖產生器的草稿或完成地圖資料，作為桌位定位與訂餐管理基礎。
- 待新模組功能驗證完整後，再以替換與淘汰方式移除舊桌位管理模組。

## 12.1 Table Edit Mode 最低互動規則

為了讓桌位編輯可直接進入實作，第一版需定義最小互動流程。

### 1. 新增桌位

- 工具列提供 `Table` 工具。
- 啟用後，點擊 `Editor Surface` 即建立一個桌位物件。
- 桌位使用預設尺寸，例如 `80 x 80`。
- 建立位置以滑鼠點擊的世界座標為準。

補充規則：

- 新建立桌位後，不自動進入 `Object Edit Mode`。
- 桌位加入 `tables` 集合，而非 `objects`。

### 2. 選取桌位

- 在 `Table Edit Mode` 下點擊桌位，進入該桌位的 `Object Edit Mode`。
- 同一時間只允許一個桌位進入編輯。

補充規則：

- 點擊地圖圖形 `polyline`、`rect`、`circle` 不應有反應。
- 命中判定只針對 `tables`。

### 3. 桌位編輯（Object Edit Mode）

進入桌位編輯後，至少支援：

- 拖曳位置 `move`
- 調整大小 `resize`
- 刪除桌位 `delete`

補充規則：

- 顯示 `bounding box` 控制點。
- 第一版不支援旋轉。
- 第一版桌位刪除可透過 `Delete` 鍵或 UI 按鈕執行。
- 刪除桌位後，系統應清除目前的 `active table`。

### 4. 刪除桌位

可透過以下方式刪除：

- `Delete` 鍵
- UI 按鈕「刪除」

刪除後：

- 自動退出 `Object Edit Mode`。

### 5. 點擊空白編輯區域

- 若有啟用 `Table` 工具，建立新桌位。
- 若沒有啟用工具，不做任何事。
- 不會退出 `Object Edit Mode`。

### 6. 模式限制

- 在 `Table Edit Mode` 下，不允許建立地圖圖形。
- 在 `Table Edit Mode` 下，不允許編輯 `objects`。
- 在桌位的 `Object Edit Mode` 下，工具列鎖定。
- 在桌位的 `Object Edit Mode` 下，不允許切換工具。

### 7. 資料結構（Table）

桌位需獨立於地圖物件：

第一版最小欄位固定為：

- `id`
- `x`
- `y`
- `width`
- `height`
- `label`

```json
{
  "id": "table_01",
  "x": 300,
  "y": 200,
  "width": 80,
  "height": 80,
  "label": "A1"
}
```

補充規則：

- 使用 world 座標。
- 第一版 `tables` 的 `render order` 以陣列順序控制。
- 陣列後方桌位顯示在上層。
- 新建立桌位預設加入陣列尾端。
- `Table Edit Mode` 仍沿用同一個 `Workspace` 與同一個 `SVG Root`。
- `tables` 應視為地圖產生器新模組內的桌位資料集合，不直接等同既有資料庫欄位結構。
- `label` 代表編輯器內的桌位顯示名稱，是否對應最終桌號，應由後續資料對接層決定。
- 第一版 `label` 可由使用者手動修改。
- 若建立桌位時未指定 `label`，系統應給預設顯示名稱。
- 第一版預設命名可採簡單連號，例如 `T1`、`T2`、`T3`。
- 預設命名僅作為編輯器內顯示用途，不直接等同最終營運桌號。
- 新模組與既有資料庫之間應透過映射或轉換層對接，不直接以舊模組資料結構反向限制新模組。
- 第一版暫不強制納入 `capacity`、`status`、`shape` 等營運欄位。
- 上述欄位保留待後續桌位營運模型階段再擴充。
- 第一版桌位物件以可視化編輯與定位為優先，不提前綁定完整營運結構。
- 進入 `Table Edit Mode` 後，使用者只操作桌位相關物件，不直接編輯地圖圖形。
- 桌位物件也可進入 `Object Edit Mode`，但其編輯內容僅限桌位自身。
- 桌位與地圖物件若同時存在於視圖中，命中與操作優先序仍需遵守模式邊界。

## 13. 地圖資料結構
為了支援地圖儲存、重新載入與後續桌位擴充，系統需採用清晰且可擴展的資料結構。

第一版資料結構分為三個層級：`Map`、`Object`、`Node`。

### 13.1 Map（地圖）
`Map` 代表一張餐廳地圖。

每張地圖包含基本屬性與所有繪圖物件。

必要欄位：

- `id`
- `name`
- `width`
- `height`
- `objects`

範例：

```json
{
  "id": 1,
  "name": "一樓內用區",
  "width": 1200,
  "height": 800,
  "objects": []
}
```

補充規則：

- `objects` 陣列順序即為 `render order`。
- 陣列後方物件顯示在上層。
- 新建立物件預設加入陣列尾端。
- 物件重疊時，命中判定優先以較上層物件為準。
- 第一版 `Map` 使用 `objects` 儲存地圖物件。
- 下一階段桌位資料建議獨立為 `tables` 集合，不直接混入 `objects`。
- 原因是桌位屬於營運物件，不等同一般場域圖形。
- 下一階段 `Map` 可擴充包含 `tables` 集合。
- `objects` 與 `tables` 為平行集合。
- `objects` 負責地圖圖形。
- `tables` 負責桌位物件。
- 不可將 `tables` 視為 `objects` 的子型別。

擴充範例：

```json
{
  "id": 1,
  "name": "一樓內用區",
  "width": 1200,
  "height": 800,
  "objects": [],
  "tables": []
}
```

- 此範例為下一階段擴充方向。
- 第一版若尚未進入桌位編輯實作，可先不強制使用此結構落地。

### 13.1.1 第一版資料落地原則
- 第一版可先以 `Map + objects` 作為地圖編輯主資料結構。
- `tables` 屬於下一階段擴充集合，不要求在地圖繪製第一階段同步落地。
- 但資料模型設計上需預留 `tables` 平行集合位置，避免後續重構 `Map` 主結構。
- 第一階段正式資料落地可僅包含 `map` 主資料與 `objects`。
- `tables` 在尚未進入桌位編輯實作前，不要求寫入正式資料。
- 但資料模型與轉換層需預留 `tables` 擴充位置。

補充原則：

- 第一版內部主資料可採 `map` 主資料 + `objects` 平行集合的形式管理。
- 若進入第二階段，可再擴充 `tables` 平行集合。
- 文件中的 JSON 範例有時將 `objects`、`tables` 放入 `map` 內，是為了描述交換格式與擴充方向。
- 實作時可依模組內部狀態管理需要，採用 `map` 主資料與物件集合分離的方式。
- 內部狀態格式與交換格式可不同，但語意必須一致，並由轉換層負責整理。

### 13.2 Object（地圖物件）
`Object` 代表 `Map Layer` 上的一個元素，例如：

- 折線
- 矩形
- 圓形
- 文字

必要欄位：

- `id`
- `type`
- `data`

範例：

```json
{
  "id": "obj_01",
  "type": "polyline",
  "data": {}
}
```

`type` 可能包含：

- `polyline`
- `rect`
- `circle`
- `text`

### 13.3 Node（節點）
`Node` 主要用於折線物件，代表一個節點座標。

節點結構：

- `x`
- `y`

範例：

```json
{
  "type": "polyline",
  "data": {
    "points": [
      { "x": 100, "y": 100 },
      { "x": 400, "y": 100 },
      { "x": 400, "y": 300 }
    ]
  }
}
```

### 13.4 其他物件資料
矩形：

```json
{
  "type": "rect",
  "data": {
    "x": 200,
    "y": 200,
    "width": 300,
    "height": 150
  }
}
```

圓形：

```json
{
  "type": "circle",
  "data": {
    "x": 220,
    "y": 120,
    "width": 160,
    "height": 160
  }
}
```

文字：

```json
{
  "type": "text",
  "data": {
    "x": 150,
    "y": 120,
    "width": 120,
    "height": 40,
    "content": "內用區"
  }
}
```

### 13.5 設計原則
第一版資料結構需遵循以下原則：

- `Map` 僅負責管理地圖尺寸與物件集合。
- `Object` 負責描述 `Map Layer` 中的圖形元素。
- `Node` 僅處理座標資料。
- 所有圖形皆可透過 `type + data` 擴展。
- 第一版圖形資料盡量統一使用矩形 `bounding box`。
- 上述作法可簡化拖曳、縮放與命中判定。
- `circle` 以外接矩形作為第一版幾何描述基礎。
- 第一版 `circle` 應維持 `width = height`，避免在資料層直接退化為橢圓。

此結構可在未來直接擴充：

- 桌位物件 `table`
- 區域物件 `zone`
- 圖層 `layer`
- 群組 `group`

而不需要重構整個資料模型。

## 14. 座標系統與縮放規則
為了支援穩定的地圖編輯、縮放顯示與後續視角平移能力，地圖系統必須明確區分「世界座標」與「畫面座標」。

### 14.1 地圖使用世界座標
地圖必須有一套固定的內部座標系統，不可直接以目前畫面顯示大小作為真實資料。

例如一張地圖可定義為：

- 地圖寬度：`1200`
- 地圖高度：`800`

所有物件都應以這組地圖原始座標儲存，例如：

- `x`
- `y`
- `width`
- `height`

設計規則如下：

- 資料永遠儲存地圖原始座標。
- 不儲存畫面縮放後的座標。
- 同一份資料在不同縮放倍率下必須保持一致。

### 14.2 縮放只影響顯示，不改變資料
當編輯視圖縮放時，改變的是畫面呈現大小，不是地圖資料本身。

例如編輯視圖放大 `150%`：

- 畫面看起來變大。
- 但資料仍維持在原本 `1200 x 800` 的世界座標系統中。

轉換公式如下：

```text
screenX = worldX * scale
screenY = worldY * scale
```

反向換算時：

```text
worldX = screenX / scale
worldY = screenY / scale
```

### 14.3 平移與縮放需分開處理
若未來要支援視角平移，座標轉換需再加入平移量。

完整公式如下：

```text
screenX = worldX * scale + offsetX
screenY = worldY * scale + offsetY
```

第一版建議如下：

- 先支援 `scale`
- 暫不支援 `offsetX / offsetY`
- 平移能力留待下一階段實作

這樣可先把資料模型與互動模型穩定下來，再擴充 `Workspace` 視角移動能力。

### 14.4 事件座標需先轉回世界座標
當滑鼠點擊或拖曳 `Editor Surface` 時，不應直接使用 `clientX / clientY` 作為地圖資料座標。

正確流程應為：

- 事件命中判定與座標換算，皆應以 `SVG Root` 的座標系統為準。
- 第一版事件命中判定應優先以 `SVG` 元素本身與其資料標記為基礎。
- 不自行模擬 HTML `<canvas>` 式命中系統。
- 座標換算仍以世界座標規則統一處理。

1. 先取得滑鼠相對於 `SVG Root` 的座標。
2. 扣掉 `Editor Surface` 在畫面上的位置。
3. 再除以目前縮放比例。
4. 轉換為世界座標後，再寫入地圖資料。

這樣可確保：

- 不同縮放倍率下新增節點位置仍然正確。
- 拖曳物件時不會因畫面縮放而產生偏移。
- 未來加入平移後，仍可沿用同一套座標轉換邏輯。

### 14.5 第一版實作原則
第一版地圖編輯器在座標系統上應遵守以下原則：

- 地圖資料一律使用世界座標。
- 畫面縮放僅影響顯示層。
- 互動事件必須先轉回世界座標再處理。
- `scale` 與未來的 `offset` 必須在設計上保留分層。
- 不可將畫面顯示座標直接寫回資料模型。

### 14.6 第一版互動命中原則
- 第一版應優先使用 `SVG` 元素本身與其資料標記進行事件命中。
- `Map Layer` 與 `Table Layer` 的命中邏輯需依當前工作模式分流。
- `Object Edit Mode` 的控制點命中應優先於一般物件命中。
- 同一位置若同時存在多個可命中元素，仍應先依模式邊界，再依圖層與 `render order` 決定優先順序。










## 15. 資料交換規格
為了讓餐廳地圖產生器、後續櫃台功能模組與既有資料庫之間能穩定對接，第一版需先定義資料交換邊界。

### 15.1 草稿儲存格式
草稿儲存用於保存目前可繼續編輯的工作狀態。

草稿資料應至少包含：

- `version`：草稿資料格式版本
- `map`：目前地圖主資料，內含目前已完成的地圖資料集合
- `draftState`：目前編輯中暫存狀態
- `savedAt`：最近一次儲存時間

範例：

```json
{
  "version": 1,
  "map": {
    "id": 1,
    "name": "一樓內用區",
    "width": 1200,
    "height": 800,
    "objects": []
  },
  "draftState": {
    "mode": "map",
    "tool": "polyline",
    "activeObjectId": null,
    "pendingPolyline": [
      { "x": 100, "y": 100 },
      { "x": 300, "y": 100 }
    ]
  },
  "savedAt": "2026-03-17T10:00:00+08:00"
}
```

補充規則：

- 草稿格式的重點是可恢復編輯狀態，而不只是保存最終結果。
- `draftState` 可包含未完成的繪製資料、目前工具、目前模式與目前選取狀態。
- 第一版若未進行桌位編輯，`map` 內可暫不包含 `tables`。
- 草稿資料也應包含 `version` 欄位，以支援未來升級與兼容處理。

### 15.2 正式儲存格式
正式儲存用於保存可穩定載入的地圖資料。

正式資料應至少包含：

- `version`：資料格式版本
- `map`：地圖主資料，內含已完成的 `objects` 與可選的 `tables`

範例：

```json
{
  "version": 1,
  "map": {
    "id": 1,
    "name": "一樓內用區",
    "width": 1200,
    "height": 800,
    "objects": [],
    "tables": []
  }
}
```

補充規則：

- 正式儲存不應依賴畫面狀態才可解讀。
- 正式儲存內容應以已完成資料為主，不強制包含編輯中暫存狀態。
- 若系統同時保留草稿與正式版本，兩者應可分開管理。
- 正式儲存範例若使用 `map` 內含 `objects`、`tables` 的格式，應視為對外交換或持久化格式。
- 內部編輯狀態不必強制與此格式完全一致。
- 資料交換格式需與第 13 章資料結構一致，不應同時維持兩套互相重疊的欄位語意。

### 15.3 櫃台模組匯入格式
後續櫃台功能模組需能匯入餐廳地圖產生器輸出的地圖資料，作為定位、訂餐與營運管理基礎。

匯入資料至少應能提供：

- 地圖基本資料
- 已完成的地圖圖形
- 已完成的桌位資料

補充規則：

- 櫃台模組匯入時，應以可穩定顯示與可定位的資料為主。
- 櫃台模組不需依賴編輯器的 `draftState` 才能顯示地圖。
- 若匯入的是草稿資料，櫃台模組應可選擇只使用其中已完成的地圖與桌位資料。
- 匯入前若需整理由草稿格式轉為櫃台可用格式，應由匯入轉換層負責。

### 15.4 與既有資料庫的映射責任
新模組與既有資料庫之間，需保留清楚的映射與轉換層責任。

補充規則：

- 餐廳地圖產生器內部資料結構不直接等同既有資料庫欄位結構。
- 與既有資料庫對接時，應由映射或轉換層負責欄位轉譯。
- 地圖圖形資料與桌位營運資料應分開映射，不混用同一組欄位語意。
- 櫃台模組若需使用桌號、狀態、訂位等營運欄位，應在匯入後由營運模型補齊，不直接回頭污染地圖編輯模型。
